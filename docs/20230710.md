# MICS 체내삽입형 의료기기 통신 시스템 설계문서

> 작성일: 2023-07-10
> MCU: STM32U575 Q-series
> RF: CC1101 (400 MHz MICS), CC2500 (2.4 GHz Wake-up)

---

## 1. 시스템 개요

체내삽입형 의료기기(Implant)와 체외 프로그래머(Programmer) 간 무선 통신 시스템.

| 구분 | 역할 | 설명 |
|------|------|------|
| **Master (Programmer)** | 체외 장치 | 통신 개시, LBT 수행, 채널 선택 |
| **Slave (Implant)** | 체내 삽입형 | 수동 응답만, 초저전력 대기 |

### 듀얼 라디오 아키텍처

| 라디오 | 주파수 | 용도 |
|--------|--------|------|
| CC2500 | 2.4 GHz ISM | Wake-up 전용 (WOR 모드) |
| CC1101 | 402~405 MHz MICS | 데이터 통신 (LBT 필수) |

> **MAC PDU 는 두 라디오 공통** — PHY(주파수·변조·싱크워드)만 다름.
> §4.3.2 의 프레임 포맷은 400 MHz 전용이 아니라 CC2500 Wake-up 비컨 /
> ACK 에도 그대로 적용되며, Master·Slave 어느 방향이든 송수신측이 같은
> `Proto_BuildPacket()` / `Proto_ParsePacket()` 을 사용합니다.
> (CC2500 링크 레벨 상세는 §4.3.1 참조.)

---

## 2. 통신 흐름

프로토콜 단계는 **Wake-up → ACK(채널 확정) → MICS 접속** 3 단계.
(Master 내부에서 비컨 송신 직전에 `CC1101_FindFreeChannel()` 로 빈 채널 N 을
골라 비컨 payload 에 실어 보낼 뿐, OTA 단계로는 드러나지 않음.)

아래 통합 시퀀스 다이어그램은 **OTA 관점의 프레임 교환 순서**를 보여준다.
§2.1 에서는 Master 와 Slave 각각의 내부 동작 흐름을 독립적인 상태 플로우로,
§2.2 에서는 두 디바이스가 공유하는 **프로토콜 스택(ISO/OSI 7 계층)** 을 정리한다.

```
Master (Programmer)                              Slave (Implant)
    |                                                |  MCU: Stop2, CC2500: WOR
    |  (내부) CC1101_FindFreeChannel → ch=N          |
    |                                                |
    |  [CC2500] Wake-up Beacon (x10 반복)            |
    |  payload = DEVICE_ID + MASTER_ID               |
    |            + SESS + CHANNEL(N) + flags  (13 B) |
    |----------------------------------------------->|
    |                                                |  GDO0 EXTI -> MCU 기상
    |                                                |  비컨 파싱 → DEVICE_ID·MASTER_ID 검증
    |                                                |  (미일치 시 silent drop)
    |                                                |  SESS/CH=N 저장, 채널 스캔 없음
    |           [CC2500] Wake-up ACK                 |
    |    payload = DEVICE_ID + MASTER_ID(에코)       |
    |              + SESS + CH(에코) + flags  (13 B) |
    |<-----------------------------------------------|  ← 채널 확정
    |  DEVICE_ID + MASTER_ID + SESS + CH echo 검증   |
    |                                                |  CC1101 wake + ch=N 직접 튜닝
    |  --- CC1101 400 MHz MICS 세션 접속 ---          |
    |  (양측 모두 ch=N 에서만 송수신)                 |
    |  (Master TX: 동일 채널 LBT 재시도만, 홉핑 금지) |
    |                                                |
    |  [CC1101] POLL_REQ  (ch=N,                     |
    |    SESS + DEVICE_ID + MASTER_ID + args)        |
    |----------------------------------------------->|
    |                                                |  SESS+DEVID+MASTERID triple verify
    |           [CC1101] POLL_RESP / DATA            |
    |    (DEVICE_ID + MASTER_ID + 센서 body)         |
    |<-----------------------------------------------|
    |                                                |
    |  [CC1101] DATA_WRITE (ch=N,                    |
    |    SESS + DEVICE_ID + MASTER_ID + 설정값)      |
    |----------------------------------------------->|
    |           [CC1101] DATA_ACK                    |
    |<-----------------------------------------------|
    |                                                |
    |  [CC1101] SLEEP_CMD                            |
    |----------------------------------------------->|
    |           [CC1101] SLEEP_ACK                   |
    |<-----------------------------------------------|
    |                                                |  CC1101: Sleep
    |                                                |  CC2500: WOR 재진입
    |                                                |  MCU: Stop2 복귀
```

**핵심:**
- **Wake-up 인증**: Slave 는 비컨의 MASTER_ID 가 공장 페어링 ID 와 일치할 때만 깨어남 — DEVICE_ID 유출만으로 웨이크업 불가 (Patch A).
- **Triple verify**: 400 MHz 모든 프레임에서 SESS + DEVICE_ID + MASTER_ID 세 가지 모두 검증, 하나라도 불일치 시 silent drop.
- **Slave 채널 스캔 없음**: 비컨에 실린 채널로 즉시 튜닝.
- **Slave TX**: `threshold=-120 dBm` 로 CCA 를 사실상 우회 (`app/slave/s_main.c` 의 `Slave_SendResponse()`).
- **Master TX (Patch B)**: 통지한 ch=N 에서만 송수신 — CCA 실패 시 **동일 채널 3회 재시도 + 랜덤 백오프**, 모두 실패하면 외곽 루프가 세션을 재시작해 **새 비컨으로 새 채널** 통지 (Slave 홉핑 불가이므로 홉 금지).

### 2.1 Master / Slave 동작 흐름 (독립 뷰)

#### 2.1.1 Master (Programmer) 동작 흐름

Master 는 **능동 측 (active side)** — 세션을 개시하고, LBT 로 채널을 선정하며, POLL/DATA/SLEEP 명령을 발행한다.

```
                        MASTER (PROGRAMMER)
                        ═══════════════════

      ┌─────────────────────────────────────────────────────┐
      │ [0]  Boot / Init                                    │
      │      · LP_Init()                                    │
      │      · CC1101_InitMICSLike26MHz()  (2-FSK, 400 MHz) │
      │      · CC2500_InitWakeUp26MHz()    (OOK,  2.4 GHz)  │
      └─────────────────────────────────────────────────────┘
                                  │
                                  ▼
      ┌─────────────────────────────────────────────────────┐
      │ [1]  LBT 채널 선정 (내부, OTA 없음)                 │
      │      CC1101_FindFreeChannel()  →  ch = N (0..9)     │
      │      g_active_channel ← N                           │
      └─────────────────────────────────────────────────────┘
                                  │
                                  ▼
      ┌─────────────────────────────────────────────────────┐
      │ [2]  CC2500 Wake-up 비컨 송신 (x10 반복 OOK)        │
      │      TX payload (13 B):                             │
      │        DTYPE_COMMAND | CMD_WAKEUP_REQ               │
      │        + DEVICE_ID(4) + MASTER_ID(4)                │
      │        + SESS(1) + CHANNEL=N(1) + flags(1)          │
      └─────────────────────────────────────────────────────┘
                                  │
                                  ▼
      ┌─────────────────────────────────────────────────────┐
      │ [3]  CC2500 Wake-up ACK 수신 & 에코 검증            │
      │      RX: DEVICE_ID · MASTER_ID · SESS · CH echo     │
      │      mismatch  →  silent drop → [2] 재시도          │
      │      timeout   →  [2] 재시도 (최대 MASTER_SESSION_  │
      │                   RESTARTS 회)                      │
      └─────────────────────────────────────────────────────┘
                                  │
                                  ▼  (CC2500 IDLE, CC1101 wake)
      ┌─────────────────────────────────────────────────────┐
      │ [4]  CC1101 ch=N 튜닝 → 400 MHz MICS 세션 진입      │
      └─────────────────────────────────────────────────────┘
                                  │
                                  ▼
      ┌─────────────────────────────────────────────────────┐
      │ [5]  POLL_REQ  loop  (센서 데이터 수집)             │
      │      TX: Master_TxLBT_SameChannel() —               │
      │          동일 채널 3회 CCA 재시도 + jitter 백오프   │
      │          (Patch B: 홉핑 금지)                       │
      │      RX: POLL_RESP → triple verify                  │
      │           (SESS + DEVICE_ID + MASTER_ID)            │
      └─────────────────────────────────────────────────────┘
                                  │
                                  ▼
      ┌─────────────────────────────────────────────────────┐
      │ [6]  DATA_WRITE  (설정값 하향)                      │
      │      TX: Master_TxLBT_SameChannel()                 │
      │      RX: DATA_ACK → triple verify                   │
      └─────────────────────────────────────────────────────┘
                                  │
                                  ▼
      ┌─────────────────────────────────────────────────────┐
      │ [7]  SLEEP_CMD  (세션 종료 알림)                    │
      │      TX: Master_TxLBT_SameChannel()                 │
      │      RX: SLEEP_ACK → triple verify                  │
      └─────────────────────────────────────────────────────┘
                                  │
                                  ▼
      ┌─────────────────────────────────────────────────────┐
      │ [8]  세션 종료                                      │
      │      CC1101 IDLE, CC2500 IDLE, MCU 대기             │
      └─────────────────────────────────────────────────────┘
                                  │
                                  ▲  다음 세션 주기   ──────┐
                                  │                          │
                                  └──────────────────────────┘
```

- **Patch B 전이**: `[5]/[6]/[7]` 에서 `CC1101_ERR_CCA` 반환 시 외곽 `Master_CommunicationSession` 루프가 `[1]` 부터 **새 LBT 채널** 로 재시작 (최대 2 회) — Slave 는 이미 종료된 세션의 비컨을 기다리지 않도록 반드시 새 비컨으로 재통지.
- **Patch A 인증**: `[3]/[5]/[6]/[7]` 모두 응답 frame 에 MASTER_ID 가 에코되어야 하며, 불일치 시 세션 파기.

#### 2.1.2 Slave (Implant) 동작 흐름

Slave 는 **수동 측 (passive side)** — 평시 Stop 2 + WOR 로 극저전력 대기하며, 비컨 수신 EXTI 로만 기상한다.

```
                        SLAVE (IMPLANT)
                        ═══════════════

      ┌─────────────────────────────────────────────────────┐
      │ [0]  Boot / Init                                    │
      │      · LP_Init()                                    │
      │      · CC1101_InitMICSLike26MHz()  → Sleep          │
      │      · CC2500_InitWakeUp26MHz()    → WOR            │
      └─────────────────────────────────────────────────────┘
                                  │
                                  ▼
  ┌──▶ ┌─────────────────────────────────────────────────────┐
  │   │ [1]  저전력 대기 (평시)                             │
  │   │      MCU    : LP_EnterStop2()       (~2 μA)         │
  │   │      CC1101 : Sleep                                 │
  │   │      CC2500 : WOR (~500 ms 스니프)                  │
  │   └─────────────────────────────────────────────────────┘
  │                               │
  │                               ▼   GDO0 EXTI (CRC_OK 비컨)
  │   ┌─────────────────────────────────────────────────────┐
  │   │ [2]  MCU 기상 / CC2500 비컨 수신                    │
  │   │      LP_RestoreFromStop2() → SPI 복원                │
  │   │      CC2500 RX payload (13 B) → Proto_ParsePacket   │
  │   └─────────────────────────────────────────────────────┘
  │                               │
  │                               ▼
  │   ┌─────────────────────────────────────────────────────┐
  │   │ [3]  Triple verify                                  │
  │   │      · Proto_VerifyPayloadDeviceID()                │
  │   │      · Proto_VerifyPayloadMasterID()                │
  │   │      mismatch → silent drop → [1] 복귀              │
  │   └─────────────────────────────────────────────────────┘
  │                               │
  │                               ▼
  │   ┌─────────────────────────────────────────────────────┐
  │   │ [4]  상태 저장 + ACK 송신                           │
  │   │      g_active_channel ← payload[CHANNEL] (0..9)     │
  │   │      g_active_sess    ← payload[SESS]               │
  │   │      TX CC2500 Wake-up ACK (MASTER_ID · CH echo)    │
  │   └─────────────────────────────────────────────────────┘
  │                               │
  │                               ▼
  │   ┌─────────────────────────────────────────────────────┐
  │   │ [5]  CC2500 → IDLE, CC1101 ch=N 직접 튜닝           │
  │   │      (채널 스캔 없음 — CH 는 비컨에서 수신)         │
  │   └─────────────────────────────────────────────────────┘
  │                               │
  │                               ▼
  │   ┌─────────────────────────────────────────────────────┐
  │   │ [6]  CC1101 RX 루프 (세션 유지 동안)                │
  │   │      프레임 수신마다 triple verify →                │
  │   │        · POLL_REQ   → TX POLL_RESP                  │
  │   │        · DATA_WRITE → TX DATA_ACK                   │
  │   │        · SLEEP_CMD  → TX SLEEP_ACK → [7]            │
  │   │      TX CCA: threshold=-120 dBm (사실상 우회)        │
  │   └─────────────────────────────────────────────────────┘
  │                               │
  │                               ▼
  │   ┌─────────────────────────────────────────────────────┐
  │   │ [7]  세션 종료 → 저전력 복귀                        │
  │   │      g_active_channel ← PROTO_CHANNEL_UNASSIGNED    │
  │   │      CC1101 Sleep, CC2500 WOR 재진입                │
  │   └─────────────────────────────────────────────────────┘
  │                               │
  └───────────────────────────────┘
```

**핵심 동작 원칙 (양측 공통):**

- **Wake-up 인증** — Slave 는 비컨 MASTER_ID 가 공장 페어링 ID 와 일치할 때만 기상 (DEVICE_ID 유출만으로 웨이크업 불가, Patch A).
- **Triple verify** — 400 MHz 모든 프레임에서 SESS + DEVICE_ID + MASTER_ID 세 가지 모두 검증, 하나라도 불일치 시 silent drop.
- **Slave 채널 스캔 없음** — 비컨에 실린 CHANNEL 필드로 즉시 튜닝 → 임플란트 전력 절감.
- **Master TX (Patch B)** — 통지한 ch=N 에서만 송수신. CCA 실패 시 동일 채널 3회 재시도, 모두 실패하면 세션을 재시작해 **새 비컨으로 새 채널** 통지 (Slave 동기화 유지).
- **Slave TX** — `threshold=-120 dBm` 로 CCA 를 사실상 우회 (`app/slave/s_main.c` 의 `Slave_SendResponse()`).

### 2.2 프로토콜 스택 / 계층 구조 (ISO/OSI 7 기반)

본 프로젝트의 소프트웨어·하드웨어를 ISO/OSI 7 계층 모델에 매핑한 계층 구조이다.
동일한 MAC PDU·빌더·파서가 두 PHY (CC1101 400 MHz / CC2500 2.4 GHz) 위에서 동작하므로 **L2 이상은 양 라디오 공유**, **L1 / L0 만 라디오별 분기** 된다.

```
                ┌─────────────────────────────────────────────────────────────┐
                │  L7  Application  ─  app/master/m_main.c · app/slave/s_main.c   │
                │                      · 세션 시나리오 FSM                    │
                │                      · POLL / DATA / SLEEP 명령 발행         │
                │                      · Triple verify 정책 집행              │
                ├─────────────────────────────────────────────────────────────┤
                │  L6  Presentation ─  common/protocol.h                      │
                │                      · MICS_DTYPE_* 부호 해석               │
                │                      · FCF 비트필드 인코딩/디코딩           │
                │                      · payload 레이아웃 (Builder/Parser)    │
                ├─────────────────────────────────────────────────────────────┤
    공통        │  L5  Session      ─  SESS(1..254) + MASTER_ID + DEVICE_ID   │
    (two        │                      · Proto_MatchSession()                 │
     radios     │                      · Proto_VerifyPayloadMasterID()        │
     share)     │                      · Proto_VerifyPayloadDeviceID()        │
                │                      · 세션 경계: Wake-up ACK ~ SLEEP_ACK   │
                ├─────────────────────────────────────────────────────────────┤
                │  L4  Transport    ─  Master_TxLBT_SameChannel()             │
                │                      · 동일 채널 3회 CCA 재시도 + 백오프    │
                │                      · 세션 재시작 외곽 루프 (최대 2회)     │
                │                      · Slave silent drop / Master echo 검증 │
                ├─────────────────────────────────────────────────────────────┤
                │  L3  Network      ─  비컨 CHANNEL 필드 기반 ch=N 프리할당   │
                │                      · Master LBT → 비컨 전달 → Slave 튜닝 │
                │                      · 400 MHz MICS 10 채널 (0..9)          │
                ├─────────────────────────────────────────────────────────────┤
                │  L2  Data Link    ─  MAC (Proto_BuildPacket / ParsePacket)  │
                │                      · PDU: FCF | SEQ | SESS | PAYLOAD | CRC│
                │                      · LLC: HW packet engine (variable-len  │
                │                              + CRC_EN + APPEND_STATUS)      │
                │                              GDO0 = CRC_OK                  │
                ├──────────────────────────────┬──────────────────────────────┤
                │  L1  Physical                │  L1  Physical                │
    분기        │     (CC1101, 400 MHz MICS)   │     (CC2500, 2.4 GHz WOR)    │
    (per-radio) │                              │                              │
                │  Modem:                      │  Modem:                      │
                │   · MDMCFG2 = 0x03 (2-FSK)   │   · MDMCFG2 = 0x33 (OOK/ASK) │
                │   · DEVIATN = 0x35           │   · DEVIATN = 0x00 (ignored) │
                │   · Rate: ~49.99 kbps        │   · Rate: ~2.4 kbps (WOR)    │
                │   · 10 ch (~299.93 kHz 간격) │   · 1 ch (WAKEUP_CHANNEL)    │
                │   · SYNC: 0xD391             │   · SYNC: 0xBEEF             │
                │                              │                              │
                │  RF:                         │  RF:                         │
                │   · FREND0, PATABLE          │   · FREND0=0x11, PATABLE 2B  │
                │   · 내부 PA / LNA            │   · 내부 PA (OOK ramp)       │
                │   · 외부 매칭 + 안테나       │   · 외부 매칭 + 안테나       │
                ├──────────────────────────────┼──────────────────────────────┤
                │  L0  Driver / Bus            │  L0  Driver / Bus            │
                │  cc1101/main/cc1101.c        │  cc2500/main/cc2500.c        │
                │   · SPI 레지스터 R/W         │   · SPI 레지스터 R/W         │
                │   · TX/RX FIFO burst         │   · TX/RX FIFO burst         │
                │   · strobe, reset, cal       │   · strobe, reset, cal, WOR  │
                ├──────────────────────────────┴──────────────────────────────┤
                │  플랫폼 서비스 (STM32U575 Q-series)                         │
                │  common/stm32u575_lowpower.c  +  HAL                        │
                │   · SPI + GPIO (CSn) + EXTI (GDO0 = CRC_OK)                 │
                │   · Stop 2 / Standby 저전력 모드, RTC wakeup                │
                │   · 클럭 (MSI/HSE), 전압 스케일링 (Range 4)                 │
                └─────────────────────────────────────────────────────────────┘
```

| 레이어 | 분류 | 구현 | CC1101 / CC2500 공유? |
|--------|------|------|:--:|
| L7 Application | 시나리오 FSM | `app/master/m_main.c`, `app/slave/s_main.c` | ✅ |

| L5 Session | SESS / MASTER_ID / DEVICE_ID triple verify | `common/protocol.h` + app | ✅ |
| L4 Transport | LBT 재시도 / 세션 재시작 / silent drop | `app/master/m_main.c` | ✅ |

| L1 Physical  | Modem (MDMCFG/DEVIATN) + RF (FREND/PATABLE) | CC1101/CC2500 레지스터 세트 | ❌ 라디오별 |
| L0 Driver / Bus | SPI/GPIO/EXTI, strobe, FIFO | `cc1101/`, `cc2500/`, `stm32u575_lowpower` | ❌ 라디오별 |

상세는 §4.3 프로토콜 섹션 (특히 §4.3.2 MAC PDU · §4.3.11 MDMCFG2 비트맵) 을 참조.

- SYSCLK: MSI 4 MHz (저전력 동작 시) ~ 최대 80 MHz
- PLL은 사용하지 않거나 옵션

**SPI1 (CC1101용)**
- Mode: Full-Duplex Master
- Baud Rate: ~5 MHz (CC1101 최대 10 MHz)
- Clock Polarity (CPOL): Low
- Clock Phase (CPHA): 1 Edge
- NSS: Disable (수동 GPIO 제어)
- Pin: PA5(SCK), PA6(MISO), PA7(MOSI)

**SPI2 (CC2500용)**
- 동일한 설정으로 SPI2 활성화
- Pin: PB13(SCK), PB14(MISO), PB15(MOSI)

**GPIO**
| 핀 | 모드 | 설명 |
|----|------|------|
| PA4 | Output Push-Pull, High | CC1101 CS |
| PB12 | Output Push-Pull, High | CC2500 CS |
| PB0 | Input, Pull-Down | CC1101 GDO0 (선택: EXTI) |
| PB1 | EXTI Mode with Rising trigger | CC2500 GDO0 (Wake-up용) |
| PC13 | Output Push-Pull | LED 표시 (선택) |

**RTC**
- Activate Clock Source ✔
- Activate Calendar ✔
- WakeUp ✔ (RTC Wake-Up Timer)
- Clock Source: LSE

**NVIC**
- EXTI line 1 (PB1, CC2500 GDO0) 인터럽트 활성화
- RTC Wake-Up Timer 인터럽트 활성화

설정 완료 후 **Project → Generate Code** (또는 Ctrl+S)

#### 3.2.3 m_main.c 통합

CubeMX가 매번 `Core/Src/main.c`를 덮어쓰므로, 아래 방식을 사용한다:

```c
/* USER CODE END Includes */

/* USER CODE END 2 */
```

체내삽입형 디바이스(Implant) 펌웨어 프로젝트 (`MICS_Slave_Implant`).

#### 3.3.2 .ioc (CubeMX) 페리페럴 설정 — 초저전력 최적화

**클럭 (저전력 우선)**
- HSE: **Disable** (절대 사용 금지)
- HSI16: Disable (필요 시에만)
- LSE: Crystal Resonator 32.768 kHz (RTC + Stop2 wake-up용)

- MSI Range: **4 MHz** (Run 시) — 더 낮은 1 MHz도 가능

- **SPI2** (CC2500): WOR 모드에서는 OFF (CC2500 자체가 알아서 RX 함)

**GPIO (Stop 2 누설 최소화 필수)**
| 핀 | 모드 | 설명 |
|----|------|------|
| PA4 | Output Push-Pull, High | CC1101 CS |
| PB12 | Output Push-Pull, High | CC2500 CS |

| 사용하지 않는 핀 | **Analog** | Stop 2 누설 최소화 |

**RTC (주기적 자가점검)**
- Activate Clock Source ✔
- Activate Calendar ✔
- **WakeUp ✔** (60초 주기)
- Clock Source: **LSE** (32.768 kHz)

**NVIC (Wake-up 소스)**
- **EXTI line 1** (PB1, CC2500 GDO0) → Enable, 우선순위 높게
- **RTC Wake-Up Timer** → Enable
- SysTick: Wake-up 후에만 Resume (저전력 모드 진입 시 Suspend)

**Power 설정**
- CubeMX에서 직접 설정 어려움 → 코드에서 `LP_Init()` 호출 시 자동 설정:

#### 4.3.1 CC2500 2.4 GHz Wake-up 프로토콜 (링크 레벨 개요)

2.4 GHz 웨이크업 라디오(CC2500)는 **임플란트(Slave)를 저전력 WOR 상태에서 깨워 400 MHz 세션으로 인도**하기 위한 용도로만 쓰인다. 데이터 통신은 전부 400 MHz(CC1101)에서 이뤄진다.

**PHY (링크 레벨):**

| 항목 | 값 |
|------|-----|
| 주파수 | 2.4 GHz ISM 대역 (CC2500 단일 채널 `CC2500_WAKEUP_CHANNEL`) |
| 변조 | **OOK (ASK)** — WOR 감도·전력 최적화 (MDMCFG2=0x33) |
| 데이터 레이트 | ~2.4 kbps (WOR 감도 최적화) |
| Sync word | 0xBEEF (400 MHz 0xD391 과 의도적 구별 — 교차 수신 방지) |
| 하드웨어 LEN/CRC | PKTCTRL0 variable-length + CRC_EN=1 |
| PA 테이블 | PATABLE[0]=0x00 (OFF '0'), PATABLE[1]=0xFE (ON '1'), FREND0.PA_POWER=1 |
| Slave 수신 모드 | WOR (~500 ms 주기 스니프), GDO0 EXTI 로 MCU 기상 |

**MAC / 애플리케이션 레이어:**

- MAC PDU 는 400 MHz 와 **완전히 동일** (§4.3.2 참조). PHY 만 다를 뿐 `Proto_BuildPacket` / `Proto_ParsePacket` 가 양쪽을 동일하게 처리.
- Wake-up 구간에서 오가는 프레임은 **정확히 2종** — `CMD_WAKEUP_REQ` (M→S) 와 `CMD_WAKEUP_ACK` (S→M).
- 비컨 PAYLOAD 는 13 B 고정 레이아웃으로 **채널 지정(400 MHz MICS 채널 0..9) + 세션 ID + Master 인증자** 를 한 번에 전달 (§4.3.6).

**방향성 (2 단계 핸드셰이크):**

```
Step 1   Master ─ CC2500 비컨 ─────────────────> Slave (WOR 수신)
         payload: [DTYPE_COMMAND | CMD_WAKEUP_REQ | DEVICE_ID | MASTER_ID
                   | SESS | CHANNEL=N | flags]      ← 총 13 B
         - DEVICE_ID: 깨울 대상 임플란트 ID (또는 브로드캐스트)
         - MASTER_ID: Master 인증자 (Slave 가 공장 페어링 ID 와 비교)
         - SESS:     이 세션에서 사용할 1B 세션 ID (Master 가 할당)
         - CHANNEL:  Master 가 LBT 로 미리 선정한 빈 400 MHz MICS 채널 (0..9)
         - 비컨은 PROTO_WAKEUP_BEACON_REPEAT (= 10) 회 반복 송신
           → Slave 의 WOR 스니프 윈도에 확실히 걸리도록

Step 2   Master <── CC2500 ACK ──────────────── Slave (기상 직후 직접 송신)
         payload: [DTYPE_COMMAND | CMD_WAKEUP_ACK | DEVICE_ID | MASTER_ID
                   | SESS | CHANNEL=N | flags]      ← 비컨과 동일 레이아웃 에코
         - DEVICE_ID: 자기 ID 로 되돌려 보냄 (송신자 증명)
         - MASTER_ID: 받은 값을 그대로 에코 (Master 가 양방향 바인딩 확인)
         - SESS:     받은 값 에코 (Master 가 자기 할당이 맞는지 확인)
         - CHANNEL:  받은 값 에코 (채널 동기화 확인 — 불일치 시 Master 가 세션 거부)
         → 이 ACK 이후부터 Slave 는 CC2500 을 끄고 CC1101 을 깨워 채널 N 에서 대기
```

**검증 정책 (양측 공통):**

- 수신한 비컨 / ACK 에서 `DEVICE_ID`·`MASTER_ID` 가 기대값과 불일치하면 **silent drop** (응답 없음)
- DEVICE_ID 만 검사하던 과거 설계는 DEVICE_ID 유출만으로 웨이크업이 가능했던 보안 공백을 Patch A 에서 MASTER_ID 를 추가해 차단
- MAC 헤더의 `SESS` 는 비컨 구간에서 `PROTO_SESS_BROADCAST` (0xFF) 로 고정 — 세션 ID 는 payload offset 10 에 실림

**왜 2.4 GHz 를 웨이크업 전용으로 쓰는가?**

1. 2.4 GHz CC2500 는 WOR 감도가 높고 MICS 대역보다 임플란트 내부 RF 누설이 적음 → 초저전력 스니프 가능 (~1.5 uA 평균).
2. 400 MHz MICS 는 인체 투과 손실이 크므로 항시 수신 모드는 전력 소모가 큼 → 데이터 전송에만 쓰고, 기상은 2.4 GHz 가 담당.
3. 기상 시 400 MHz 채널 번호를 **미리 실어 보내면** Slave 의 채널 스캔이 필요 없어져 임플란트 배터리 추가 절감.

#### 4.3.2 공통 애플리케이션 PDU (CC1101 전 구간 + CC2500 Proto_BuildPacket 구간)

```
+------+------+------+------+------------------+------+
| LEN  | FCF  | SEQ  | SESS | PAYLOAD          | CRC  |
| 1B   | 1B   | 1B   | 1B   | 0..58B (no FEC)  | 2B   |
|      |      |      |      | 0..105B (w/FEC)  |      |
+------+------+------+------+------------------+------+
      └─────── MAC HEADER (3B) ─────┘
```

- **LEN / CRC**: CC1101·CC2500 하드웨어가 자동 처리 (PKTCTRL0.LENGTH_CONFIG=variable, CRC_EN=1)
- **MAC 프레임은 두 라디오 공통** — PHY(주파수·변조·싱크워드)만 다를 뿐 PDU 구조는 동일
- **DEVICE_ID(4B) + MASTER_ID(4B)는 MAC 헤더에 없음** — SESS(1B)가 세션 식별 담당. DEVICE_ID·MASTER_ID 는 모든 PAYLOAD(비컨 / COMMAND / DATA) 내부에 실려 핸드오프·인증에 사용
- **MASTER_ID**: Slave 가 공장 페어링된 Master 만 수용하기 위한 4B 인증자 (Patch A). 비컨 / COMMAND / DATA 모든 프레임 PAYLOAD 에 포함되어 수신측이 `Proto_VerifyPayloadMasterID()` 로 검증
- **400 MHz CHANNEL 번호도 Wake-up 비컨 PAYLOAD(§4.3.6 offset 11)에 실림** — Master(송신측)가 LBT로 사전 선정, Slave(수신측)가 그대로 채택. 양측 공통 필드이며 `Proto_BuildBeaconPayload()` / `Proto_ParsePacket()` 한 쌍으로 처리됨

| 구간 | 라디오 | MAC PDU | 빌더/파서 |
|------|--------|---------|----------|
| Master→Slave Wake-up 비컨 | CC2500 (2.4 GHz) | 동일 | `Proto_BuildPacket` / `Proto_BuildBeaconPayload` |
| Slave→Master Wake-up ACK | CC2500 (2.4 GHz) | 동일 | 동일 |
| Master→Slave POLL/WRITE/SLEEP | CC1101 (400 MHz) | 동일 | `Proto_BuildPacket` / `Proto_BuildCommandPayload` |
| Slave→Master 데이터/ACK | CC1101 (400 MHz) | 동일 | `Proto_BuildPacket` / `Proto_BuildDataPayload` |

| 필드 | 크기 | 설명 |
|------|------|------|
| LEN | 1 B | 프레임 길이 (헤더 포함, 최대 63 B) — 라디오 자동 |
| FCF | 1 B | 프레임 제어 필드 (비트 구조 참조) |
| SEQ | 1 B | 시퀀스 번호 (0~255 wrap) |
| SESS | 1 B | 세션 ID (0x01~0xFE 할당, 0x00=미할당, 0xFF=브로드캐스트) |
| PAYLOAD | 0~58 B (FEC 미적용) / 0~105 B (FEC 적용) | 데이터 페이로드 |
| CRC | 2 B | 라디오 자동 |

#### 4.3.3 FCF (Frame Control Field) 비트 구조

| 비트 | 매크로 | 설명 |
|------|--------|------|
| Bit 7 (0x80) | `MICS_FCF_ACK_REQ` | ACK 요청 플래그 (1: ACK 필요, 0: ACK 불필요) |
| Bit 6 (0x40) | `MICS_FCF_FRAGMENT` | Fragment 패킷 플래그 (1: Fragment, 0: 일반) |
| Bit 5 (0x20) | Reserved | 예약 |
| Bit 4 (0x10) | Reserved | 예약 |
| Bit 3~0 (0x0F) | 프레임 타입 | `MICS_FCF_TYPE_DATA` (0x00), `MICS_FCF_TYPE_ACK` (0x01), `MICS_FCF_TYPE_COMMAND` (0x02) |

#### 4.3.4 Payload 타입 코드 (PAYLOAD[0] @ DATA/COMMAND frame)

| 코드 | 매크로 | 설명 |
|------|--------|------|
| 0x10 | `MICS_DTYPE_ECG_DELTA` | ECG 델타 압축 데이터 |
| 0x20 | `MICS_DTYPE_BATTERY` | 배터리 상태 |
| 0x30 | `MICS_DTYPE_TEMP` | 온도 데이터 |
| 0x40 | `MICS_DTYPE_STATUS_LOG` | 상태 로그 |
| 0xA0 | `MICS_DTYPE_COMMAND` | 제어 명령 |
| 0xFF | `MICS_DTYPE_EMERGENCY` | 긴급 데이터 (우선 전송) |

#### 4.3.5 Command 서브코드 (PAYLOAD[1] @ DTYPE_COMMAND frame)

| 코드 | 매크로 | 방향 | 라디오 | 설명 |
|------|--------|------|--------|------|
| 0x01 | `CMD_WAKEUP_REQ` | M→S | CC2500 | Wake-up 비컨 (DEVICE_ID + MASTER_ID + SESS + CHANNEL 포함) |
| 0x02 | `CMD_WAKEUP_ACK` | S→M | CC2500 | Wake-up 응답 (DEVICE_ID + MASTER_ID + SESS + CHANNEL 에코) |
| 0x10 | `CMD_POLL_REQ` | M→S | CC1101 | 데이터 요청 |
| 0x11 | `CMD_POLL_RESP` | S→M | CC1101 | 데이터 응답 (또는 DATA 프레임 직접 응답) |
| 0x20 | `CMD_DATA_WRITE` | M→S | CC1101 | 설정값 쓰기 |
| 0x21 | `CMD_DATA_ACK` | S→M | CC1101 | 쓰기 확인 |
| 0x30 | `CMD_SLEEP_CMD` | M→S | CC1101 | 슬립 명령 |
| 0x31 | `CMD_SLEEP_ACK` | S→M | CC1101 | 슬립 확인 |
| 0x40 | `CMD_KEEPALIVE` | M→S | CC1101 | 세션 연장 |
| 0x41 | `CMD_KEEPALIVE_ACK` | S→M | CC1101 | 세션 연장 확인 |

#### 4.3.6 Wake-up 비컨 PAYLOAD 레이아웃 (CC2500, 2.4 GHz)

```
Offset  Bytes  Field
------  -----  ----------------------------------------
  0      1    MICS_DTYPE_COMMAND (0xA0)
  1      1    CMD_WAKEUP_REQ (0x01)  or  CMD_WAKEUP_ACK (0x02)
  2..5   4    DEVICE_ID  (target or responder)
  6..9   4    MASTER_ID  (Master 인증자 — Slave 가 공장 페어링 ID 와 비교)
  10     1    SESS       (Master가 할당한 세션 ID)
  11     1    CHANNEL    (Master가 LBT로 사전 선정한 MICS 채널 0..9)
  12     1    flags / reserved
                                      총 PAYLOAD 길이 = 13 B
```

- MAC SESS 필드 = `PROTO_SESS_BROADCAST` (0xFF) — 세션이 아직 성립되지 않았으므로
- 할당된 SESS는 PAYLOAD offset 10 에 실림 → Slave가 기억 → 이후 MICS 통신에서 MAC SESS로 사용
- MASTER_ID(offset 6..9)는 Slave 가 공장 페어링된 `g_authorized_master_id` 와 비교하여 불일치 시 **silent drop**. DEVICE_ID 만으로는 방어 못 하는 "DEVICE_ID 를 아는 공격자의 Wake-up" 을 차단
- Slave 는 WAKEUP_ACK 에 MASTER_ID 를 **에코** → Master 는 자기 ID 가 돌아온 것을 확인하여 양방향 바인딩 성립
- CHANNEL(offset 11)은 Master가 **비컨 송신 직전**에 400 MHz LBT로 찾은 빈 MICS 채널
  - Slave는 해당 채널로 **바로 튜닝**해서 대기 → 10개 채널 스캔 없음 → 임플란트 전력 절감
  - Slave는 ACK에 동일한 채널을 **에코**하여 Master가 채널 동기화를 확인
  - 미할당 시 `PROTO_CHANNEL_UNASSIGNED` (0xFF) — 권장되지 않음

#### 4.3.7 400 MHz COMMAND 프레임 PAYLOAD 레이아웃 (FCF type=0x02)

```
Offset  Bytes  Field
------  -----  ----------------------------------------
  0      1    MICS_DTYPE_COMMAND (0xA0)
  1      1    sub-command (CMD_POLL_REQ, CMD_DATA_WRITE, ...)
  2..5   4    DEVICE_ID  ← 2.4 GHz 웨이크업에서 넘어온 Implant ID
  6..9   4    MASTER_ID  ← 2.4 GHz 웨이크업에서 넘어온 Master 인증자
  10..   N    command-specific arguments
                                      헤더 고정 = 10 B
```

- DEVICE_ID(offset 2..5)와 MASTER_ID(offset 6..9)는 **방향과 무관하게 모든 COMMAND 프레임에 존재**
- Master → Slave: "나는 2.4 GHz 에서 너를 깨운 그 Master 0xXXXX 이고, 너 0xYYYY 와 통신 중이다"
- Slave → Master: "나는 네가 깨운 그 Implant 0xYYYY 가 맞고, 나는 네 Master ID 0xXXXX 에 바인딩돼 있다"
- 수신측은 SESS + DEVICE_ID + MASTER_ID **세 가지 모두** 검증 (§4.3.9)

#### 4.3.8 400 MHz DATA 프레임 PAYLOAD 레이아웃 (FCF type=0x00)

```
Offset  Bytes  Field
------  -----  ----------------------------------------
  0      1    data type (MICS_DTYPE_ECG_DELTA, _BATTERY, _TEMP, ...)
  1..4   4    DEVICE_ID  ← Implant 의 자기 ID (송신자 증명)
  5..8   4    MASTER_ID  ← 바인딩된 Master 인증자 에코
  9..    N    telemetry / data body
                                      헤더 고정 = 9 B
```

#### 4.3.9 2.4 GHz → 400 MHz 식별자 핸드오프 & 인증 (Triple Verify)

**Master 및 Slave 양측이 모든 400 MHz 패킷에서 다음 세 조건을 **모두** 검사:**

| 검증 항목 | 필드 | 크기 | 속도 | 역할 |
|---|---|---|---|---|
| 세션 필터 | MAC.SESS | 1 B | 빠름 | 빠른 1차 필터 (세션 불일치는 즉시 drop) |
| 피어 신원 | payload.DEVICE_ID | 4 B | O(4) | 2.4 GHz 핸드오프 바인딩 (Implant 확인) |
| Master 인증 | payload.MASTER_ID | 4 B | O(4) | 공장 페어링된 Master 인증자 (Patch A) |

```
[2.4 GHz wake-up]                         [400 MHz MICS session]
 M ── DEVICE_ID + MASTER_ID + SESS ─> S   M ── SESS + DEVICE_ID + MASTER_ID ─> S
 M <─ DEVICE_ID + MASTER_ID + SESS ── S   M <─ SESS + DEVICE_ID + MASTER_ID ── S
      │                                         │         │            │
      └── 2.4 GHz 에서 공유 ──────────────────┴─────────┴────────────┘
              ↓
     양측 모두 수신 패킷의
       SESS       == g_session_id           ← 1B 필터
       DEVICE_ID  == expected (my_id / target_id)  ← 핸드오프 검증 (4B)
       MASTER_ID  == expected (authorized_master_id / g_master_id)  ← 인증 (4B)
```

- **Master**: `payload.DEVICE_ID == g_target_id` 그리고 `payload.MASTER_ID == g_master_id` 확인
- **Slave**: `payload.DEVICE_ID == g_my_id` 그리고 `payload.MASTER_ID == g_authorized_master_id` 확인
- 어느 하나라도 불일치 시 **silent drop** (무응답) — 스푸퍼 · 다른 Master · 오착신 방어
- 헬퍼:
  - `Proto_VerifyPayloadDeviceID(pkt, offset, expected)` — COMMAND:offset=2, DATA:offset=1
  - `Proto_VerifyPayloadMasterID(pkt, offset, expected)` — BEACON:offset=6, COMMAND:offset=6, DATA:offset=5
- ⚠️ MASTER_ID 는 **공장 사전 공유 식별자**이며 암호학적 키가 아님. 재생 공격(replay) · 강한 인증이 필요하면 상위 계층에서 HMAC / nonce / 서명을 덧붙여야 함

#### 4.3.10 CC1101 vs CC2500 MAC 동일성

| 레이어 | CC1101 (400 MHz MICS) | CC2500 (2.4 GHz Wake-up) | 공통? |
|--------|----------------------|-------------------------|-------|
| PHY (변조/레이트) | 2-FSK 50 kbps, MICS ch0~9 | OOK 2.4 kbps, ch0 | ❌ |
| Sync Word | 0xD391 | 0xBEEF (의도적 구별) | ❌ |
| 하드웨어 LEN/CRC | PKTCTRL0 variable+CRC_EN | 동일 | ✅ |
| **MAC PDU** | FCF\|SEQ\|SESS\|PAYLOAD | 동일 | ✅ **Yes** |
| Builder/Parser | `Proto_BuildPacket`/`Proto_ParsePacket` | 동일 | ✅ |

**결론**: 두 라디오는 PHY만 다르고 MAC PDU는 완전히 동일. 단일 `Proto_Packet` 구조와 단일 빌더/파서가 양쪽을 모두 처리함.

#### 4.3.11 MDMCFG2 레지스터 비트맵 참조 (CC1101 · CC2500 공통)

CC1101 / CC2500 의 `MDMCFG2` 는 변조 방식과 sync-word 모드를 정하는 핵심 PHY 레지스터. 두 칩의 비트 레이아웃은 **동일**하며, 본 프로젝트에서는 다음과 같이 칩별로 다른 값을 사용한다.

**비트 레이아웃**

| 비트 | 필드 | 설명 |
|------|------|------|
| 7 | `DEM_DCFILT_OFF` | DC blocking 필터 disable (0 = enabled) |
| 6:4 | `MOD_FORMAT` | 변조 방식 (아래 표 참조) |
| 3 | `MANCHESTER_EN` | Manchester 인코딩 enable |
| 2:0 | `SYNC_MODE` | Sync-word 검출 모드 |

**`MOD_FORMAT[6:4]` 값 (CC2500 Table 28 / CC1101 동일)**

| 값 (bin) | 변조 | CC1101 | CC2500 |
|----------|------|:------:|:------:|
| `000` | 2-FSK | ✅ | ✅ |
| `001` | GFSK | ✅ | ✅ |
| `011` | ASK / OOK | ✅ | ✅ |
| `100` | 4-FSK | ✅ (E1 ONLY) | — |
| `111` | MSK | ✅ | ✅ |

**본 프로젝트 설정**

| 칩 | 초기화 함수 | `MDMCFG2` | 해석 | `DEVIATN` |
|----|-------------|-----------|------|-----------|
| **CC1101** (400 MHz MICS) | `CC1101_InitMICSLike26MHz()` | `0x03` = `0b0000_0011` | MOD_FORMAT=`000` (**2-FSK**) · SYNC_MODE=`011` (30/32) | `0x35` (~20.63 kHz, 유효) |
| **CC2500** (2.4 GHz WOR) | `CC2500_InitWakeUp26MHz()` | `0x33` = `0b0011_0011` | MOD_FORMAT=`011` (**OOK/ASK**) · SYNC_MODE=`011` (30/32) | `0x00` (OOK 에서 무시) |

> **참고**
> - OOK 모드에서는 `DEVIATN` 이 하드웨어에서 무시되므로 0 으로 둔다.
> - OOK 는 `PATABLE[0]` = '0' 상태 (carrier OFF), `PATABLE[1]` = '1' 상태로 2-엔트리 PA ramp 를 사용하며, `FREND0.PA_POWER=1` 로 인덱스 1 을 선택해야 한다 (§4.3.1 PA 테이블 참조).
> - 2-FSK 모드에서는 `DEVIATN` 이 유효 (deviation = f_XOSC · (8+DEVIATN_M)·2^DEVIATN_E / 2^17). `0x35` = DEVIATN_E=3, DEVIATN_M=5 → ~20.63 kHz.

```
│   │
│   ├── [Phase 2] Master_PollImplantData()   // 채널은 이미 확정됨
│   │   ├── Master_TxLBT_SameChannel()     // ★ 동일 채널 CCA 재시도 (홉핑 금지)
│   │   │   ├── CC1101_SendPacketLBT() × MASTER_SAME_CH_RETRIES
│   │   │   ├── 랜덤 백오프 (MASTER_SAME_CH_BACKOFF_MS + jitter)
│   │   │   └── 실패 지속 → CC1101_ERR_CCA 반환 → 외곽 루프가 세션 재시작
│   │   └── CC1101_WaitAndReadPacket()     // POLL_RESP 수신
│   │       └── SESS + DEVICE_ID + MASTER_ID triple verify
│   │
│   ├── [Phase 2] Master_WriteToImplant()
│   │   ├── Master_TxLBT_SameChannel()     // DATA_WRITE 전송 (동일 채널 재시도)
│   │   └── CC1101_WaitAndReadPacket()     // DATA_ACK 수신 (triple verify)
│   │
│   └── [Phase 3] Master_SendSleepCommand()
│       ├── Master_TxLBT_SameChannel()     // SLEEP_CMD 전송 (동일 채널 재시도)
│       └── CC1101_WaitAndReadPacket()     // SLEEP_ACK 수신 (선택)
│
└── Radios -> Sleep
```

**LBT + 채널 선정 알고리즘 (Phase 1 에서 *비컨 전* 수행):**
1. 10개 MICS 채널 순차 스캔 (CCA)
2. RSSI < -80 dBm 인 채널 발견 시 → `g_active_channel` 에 저장
3. 모든 채널 사용중 → 세션 포기 (`rc = -1`), 이후 재시도
4. 선정된 채널을 Wake-up 비컨 payload offset 11 에 삽입 → Slave 가 스캔 없이 바로 튜닝

**동일 채널 LBT 재시도 (Patch B, Phase 2 전 구간):**
- 비컨에서 채널 N 을 이미 Slave 에게 통지했으므로 **채널 홉핑 금지** — 홉하면 Slave 가 동기화에 실패
- `Master_TxLBT_SameChannel()` 이 동일 채널에서 CCA 를 `MASTER_SAME_CH_RETRIES` 회 반복
- 매 시도 사이에 `MASTER_SAME_CH_BACKOFF_MS + jitter(0~7 ms)` 백오프 → 경쟁 Master 디코럴레이션
- 모든 재시도 실패 시 `CC1101_ERR_CCA` 반환 → 외곽 루프가 세션 재시작

**외곽 세션 재시작 (Patch B, `Master_CommunicationSession`):**
- Phase 2 에서 `CC1101_ERR_CCA` 발생 → 통지했던 채널이 세션 중에 막힘
- `for (attempt = 0..MASTER_SESSION_RESTARTS)` 루프가 Phase 1 (Wake-up) 을 다시 실행
- Wake-up 이 매번 **새로운 LBT** 를 돌려 빈 채널을 다시 선정 → 새 채널을 비컨으로 전달
- Slave 는 세션 타임아웃(10 s) 후 다시 웨이크업 가능 또는 새 비컨으로 즉시 재동조
- 모든 재시작 실패 시 (`-rc != 0`) 세션 포기 → 상위에서 다음 트리거 대기

**Slave 핵심 원칙:**
- **스스로 통신을 개시하지 않음** - 오직 Master의 요청에 응답
- **채널 스캔 없음** - Master가 LBT로 선정한 채널(비컨 offset 11)으로 바로 튜닝 → 전력 절감
- **Master 인증** - MASTER_ID 가 공장 페어링 ID (`g_authorized_master_id`) 와 일치하지 않으면 **silent drop** — DEVICE_ID 유출만으로는 임플란트를 깨울 수 없음
- **세션 타임아웃** - 10초 무응답 시 자동 슬립 복귀

## 7. 전력 소비 예상

> **단위 표기 규칙**
> - **순시 소비전류 (Instantaneous current)**: `uA` / `mA` — 해당 모드에서 흐르는 전류
> - **시간 평균 소비 (Time-averaged consumption)**: `uA·h` / `mAh` — 1시간 누적 사용량
> - "통신 세션 중 ~20 mA" 는 **세션이 활성화된 동안의 순시 전류**이며,
>   1시간 내내 이 값이 흐른다는 의미가 아님 (`mAh` 아님).

### 7.1 Slave (Implant) - 일반 대기 상태 (주 동작 모드)

| 구성요소 | 모드 | 순시 소비전류 |
|----------|------|--------------:|
| STM32U575 | Stop 2 | ~2 uA |
| CC2500 | WOR (~500ms 주기 sniff) | ~1.5 uA 평균 |
| CC1101 | SPWD (Sleep) | ~0.2 uA |
| **합계 (Idle)** | **대기 시** | **~3.7 uA** |

### 7.2 Slave (Implant) - 통신 세션 중 (피크 전류, 단발성)

| 구성요소 | 모드 | 순시 소비전류 |
|----------|------|--------------:|
| STM32U575 | Run (MSI 4 MHz) | ~2 mA |
| CC1101 | RX / TX | ~15~17 mA |
| CC2500 | Idle (세션 중 OFF 가능) | ~1.5 mA |
| **합계 (Active 피크)** | **세션 진행 중에만** | **~18~20 mA** |

> ⚠️ 위의 ~20 mA 값은 **세션이 진행 중일 때의 순시(peak) 전류**입니다.
> 1시간 동안 흐르는 양(`mAh`)이 아닙니다.

### 7.3 Slave (Implant) - 시간 평균 소비량 (1시간 기준, mAh)

세션 길이와 빈도에 따른 1시간당 실제 소비 누적량 예시:

| 세션 길이 | 세션 빈도 | 활성 시간/시간 | Active 누적 | Idle 누적 | **총 소비/시간** |
|----------:|----------:|---------------:|------------:|----------:|-----------------:|
| 0.5 s | 1회/시간 | 0.5 s (0.014%) | 0.0028 mAh | 0.0037 mAh | **~0.0065 mAh/h** |
| 1.0 s | 1회/시간 | 1.0 s (0.028%) | 0.0056 mAh | 0.0037 mAh | **~0.0093 mAh/h** |
| 1.0 s | 6회/시간 | 6.0 s (0.17%) | 0.033 mAh | 0.0037 mAh | **~0.037 mAh/h** |
| 1.0 s | 60회/시간 | 60 s (1.67%) | 0.33 mAh | 0.0037 mAh | **~0.34 mAh/h** |

> 예: **세션 1초/시간 사용 시 ≈ 0.0093 mAh/h ≈ 0.22 mAh/일 ≈ 6.7 mAh/월**
> CR2032 (~225 mAh) 기준 약 **2~3년 수명 가능** (자가방전 미고려).

### 7.4 Master (Programmer / External Device) - 전력 소비

체외 장치는 일반적으로 **유선 충전 또는 대용량 배터리** 사용. Implant만큼 엄격한 저전력 제약 없음.

#### 7.4.1 Master - Idle (사용자 트리거 대기)

| 구성요소 | 모드 | 순시 소비전류 |
|----------|------|--------------:|
| STM32U575 | Stop 2 (또는 LP Run) | ~2 uA ~ 50 uA |
| CC2500 | SPWD (Sleep) | ~0.4 uA |
| CC1101 | SPWD (Sleep) | ~0.2 uA |
| 사용자 인터페이스 (LCD/버튼) | 표시 OFF | ~10 uA ~ 1 mA |
| **합계 (Idle)** | **대기 시** | **~12 uA ~ 1 mA** |

#### 7.4.2 Master - Wake-up Beacon 전송 중 (CC2500)

| 구성요소 | 모드 | 순시 소비전류 |
|----------|------|--------------:|
| STM32U575 | Run (MSI 4 MHz) | ~2 mA |
| CC2500 | TX (+1 dBm, 비컨 반복) | ~21 mA |
| CC1101 | Sleep | ~0.2 uA |

> 비컨 1회 송출 시간: ~5 ms × 10회 반복 + 간격 = **~150 ms 정도**

#### 7.4.3 Master - LBT + MICS 데이터 통신 중 (CC1101)

| 구성요소 | 모드 | 순시 소비전류 |
|----------|------|--------------:|

| CC1101 | RX (LBT/CCA) | ~15~17 mA |
| CC1101 | TX (-16 dBm, MICS 한도) | ~13~14 mA |

| **합계 (TX 피크)** | **데이터 송신 중** | **~15~16 mA** |

#### 7.4.4 Master - 세션 평균 소비량 (1시간 기준)

| 시나리오 | 세션 시간 | 세션 빈도 | **총 소비/시간** |

> Master는 일반적으로 **300~2000 mAh 배터리** 또는 **USB 전원** 사용 → 수일~수개월 운용 가능.

## 8. MICS 규정 준수 사항

- **LBT (Listen Before Talk)**: Master는 전송 전 반드시 CCA 수행
- **최대 TX 출력**: MICS 대역 25 uW (-16 dBm) EIRP
- **채널 점유 시간**: 최소화 (짧은 패킷, 빠른 세션)
- **채널 어질리티**: 채널 사용중 시 랜덤 백오프 + 채널 전환
- **Slave 수동성**: 체내 기기는 스스로 전송 개시 금지

### 11.1 f019e91 상세 (MAC PDU 재설계)

- **이전 (6B 헤더)**: `LEN | CMD | SEQ | DEVICE_ID[4] | PAYLOAD(≤55) | CRC`
- **이후 (3B 헤더)**: `LEN | FCF | SEQ | SESS | PAYLOAD(≤58 no-FEC / ≤105 FEC) | CRC`
- **FCF 비트 구조 도입**: ACK_REQ(Bit7) / FRAGMENT(Bit6) / FrameType(Bit3-0: DATA/ACK/COMMAND)
- **Payload 타입 코드**: `MICS_DTYPE_ECG_DELTA` (0x10), `BATTERY` (0x20), `TEMP` (0x30), `STATUS_LOG` (0x40), `COMMAND` (0xA0), `EMERGENCY` (0xFF)
- **SESS 기반 주소지정**: Master가 Wake-up 시 1~254 rolling 할당 → Slave가 기억 → MICS 세션 전 구간 `Proto_MatchSession()` 으로 필터링
- **DEVICE_ID(4B)**: MAC 헤더에서 제거, Wake-up 비컨 PAYLOAD offset 2~5 에만 포함
- **드라이버 계층 정리**: `CC2500_SendWakeUpBeacon(dev, device_id, id_len, repeat)` → `CC2500_SendWakeUpBeacon(dev, pkt, pkt_len, repeat)` — 드라이버에서 PDU 조립 코드 제거, MAC PDU 조립은 `Proto_BuildPacket`/`Proto_BuildBeaconPayload` 가 단독 담당
- **MAC 단일화 확인**: CC1101과 CC2500은 PHY만 다르고 (주파수·변조·싱크워드) MAC PDU와 빌더·파서는 완전히 동일함을 §4.3.10 표에 명문화

변경 파일 (6):
`common/protocol.h`, `cc2500/header/cc2500.h`, `cc2500/main/cc2500.c`, `app/master/m_main.c`, `app/slave/s_main.c`, `docs/20230710.md`

### 11.2 비컨 채널 프리할당 상세

**문제**: Master가 LBT로 빈 MICS 채널을 선정해도 Slave에게 알려줄 방법이 없으면,
Slave는 10개 채널(0~9)을 반복 스캔해야 함 → 임플란트 배터리 낭비.

**해결 방법 1**: 비컨 payload를 1바이트 확장해서 선정 채널을 실어 보냄.

**프로토콜 변경**:
- `PROTO_BEACON_PAYLOAD_LEN` 8 → 9
- `PROTO_BEACON_CHANNEL_OFFSET = 7` (신규)
- `PROTO_BEACON_FLAGS_OFFSET` 7 → 8
- `PROTO_CHANNEL_UNASSIGNED = 0xFF` (신규 — 미할당 마커)
- `Proto_BuildBeaconPayload()` 시그니처에 `uint8_t channel` 인자 추가

**Master 동작 변경** (`app/master/m_main.c` `Master_WakeUpImplant()`):
1. **비컨 송신 *전*** 에 `CC1101_FindFreeChannel()` 호출
2. 반환된 채널을 `g_active_channel` 에 저장
3. `Proto_BuildBeaconPayload(..., channel=g_active_channel, ...)` 로 비컨에 포함
4. ACK 수신 시 `payload[7] == g_active_channel` 확인 → 불일치 시 세션 거부

**Slave 동작 변경** (`app/slave/s_main.c`):
- `g_active_channel` 상태 변수 추가 (초기값 `PROTO_CHANNEL_UNASSIGNED`)
- `Slave_HandleWakeUp()`: 비컨 payload[7] 범위 검증 후 `g_active_channel` 저장, ACK에 에코
- `Slave_HandleMICSSession()`: `for (ch=0..9)` 스캔 루프 제거, `g_active_channel` 단일 채널로 직접 튜닝
- 세션 종료 시 `g_active_channel` 을 `PROTO_CHANNEL_UNASSIGNED` 로 초기화

**전력 이득**: Slave가 채널당 50 ms × 10개 = 500 ms/루프 스캔 → 단일 채널 100 ms 윈도 (~5배 절감).

### 11.3 Patch A + B 상세 (Master 인증 & 동일 채널 LBT 재시도)

#### 문제 A — Master 인증 부재 (Patch A)

이전 설계에서는 Slave 가 수신한 Wake-up 비컨 / COMMAND 프레임에서 **DEVICE_ID 만** 검증했다.
그러나 DEVICE_ID 는 공장에서 인쇄되거나 페어링 과정에서 평문 노출될 수 있는 **피어 식별자**이지 인증자가 아니다.

- 공격 시나리오: DEVICE_ID 를 아는 공격자가 임의 MASTER 역할로 Wake-up 비컨을 전송 → Slave 가 깨어나서 세션을 맺음
- 결과: 임플란트 배터리 소모 / 데이터 누수 / 제어 명령 오동작 가능

**해결 — MASTER_ID(4B) 를 모든 프레임 payload 에 추가하고 양측에서 검증:**

| 프레임 | 이전 | 이후 | 추가 필드 위치 |
|---|---|---|---|
| Wake-up 비컨 payload | 9 B | 13 B | offset 6..9 = MASTER_ID |
| 400 MHz COMMAND 헤더 | 6 B | 10 B | offset 6..9 = MASTER_ID |
| 400 MHz DATA 헤더 | 5 B | 9 B | offset 5..8 = MASTER_ID |

**빌더/파서 시그니처 변경**:
```
Proto_BuildBeaconPayload(cmd, device_id, /* NEW */ master_id, sess, channel, flags, out, out_size);
Proto_BuildCommandPayload(sub_cmd, device_id, /* NEW */ master_id, args, args_len, out, out_size);
Proto_BuildDataPayload(dtype, device_id, /* NEW */ master_id, body, body_len, out, out_size);
bool Proto_VerifyPayloadMasterID(pkt, masterid_offset, expected_id);   // NEW helper
bool Proto_MatchMasterID(id_a, id_b);                                   // NEW helper
```

**Master 측 (`app/master/m_main.c`):**
- `static const uint8_t g_master_id[4] = {0xAA, 0xBB, 0xCC, 0xDD};` 추가
- 비컨 / POLL / DATA_WRITE / SLEEP 모든 TX 에 `g_master_id` 삽입
- 비컨 ACK, POLL 응답, DATA_ACK 수신 시 `Proto_VerifyPayloadMasterID()` 로 자기 ID 에코 검증

**Slave 측 (`app/slave/s_main.c`):**
- `static const uint8_t g_authorized_master_id[4] = {0xAA, 0xBB, 0xCC, 0xDD};` (공장 페어링)
- `Slave_HandleWakeUp()`: 비컨 수신 직후 MASTER_ID 검증 → 불일치 시 **silent drop**
- `Slave_HandleMICSSession()`: 매 COMMAND 수신마다 SESS + DEVICE_ID + MASTER_ID triple verify
- 응답 (ACK / POLL_RESP / DATA_ACK) payload 에도 `g_authorized_master_id` 를 에코하여 Master 가 양방향 바인딩 확인

⚠️ **보안 한계**: MASTER_ID 는 공장 사전 공유 식별자이며 암호학적 키가 아니다. 재생 공격 · 강한 위조 방지가 필요하면 상위 계층에서 HMAC / 서명 / 시도-응답 nonce 를 덧붙여야 한다.

#### 문제 B — 통지한 채널에서 LBT 실패 (Patch B)

이전 설계에서는 Phase 2 TX 에 `CC1101_SendPacketLBT_Agile()` 을 사용했다. 이 함수는 CCA 실패 시
**다른 채널로 홉핑**한다. 그러나 Master 는 이미 Wake-up 비컨으로 "채널 N 에서 만나자" 라고 Slave 에게 통지했고,
Slave 는 채널 N 에만 귀 기울이고 있다. Master 가 채널 N→M 으로 홉하면 **Slave 가 영영 수신하지 못한다**.

**해결 — 두 단계 재시도 전략:**

1. **안쪽 루프 — 동일 채널 CCA 재시도** (`Master_TxLBT_SameChannel()`):
   - 채널 N 에서 `CC1101_SendPacketLBT()` 를 `MASTER_SAME_CH_RETRIES` (= 3) 회 반복
   - 매 시도 사이 `MASTER_SAME_CH_BACKOFF_MS` (= 5 ms) + jitter(0~7 ms) 백오프
   - jitter = `(g_seq + attempt) & 0x07` — 경쟁 Master 간 디코럴레이션
   - 모두 실패 → `CC1101_ERR_CCA` 반환 (홉하지 않음)

2. **바깥쪽 루프 — 세션 재시작** (`Master_CommunicationSession()`):
   - Phase 2 에서 `CC1101_ERR_CCA` 를 받으면 세션 전체를 재시작
   - `for (attempt = 0..MASTER_SESSION_RESTARTS)` (최대 2 회 재시작 = 총 3회 시도)
   - 재시작 시 Phase 1 Wake-up 이 새 LBT 를 돌려 **새로운 빈 채널** 을 선정 → 새 비컨으로 Slave 재동조
   - 비 CCA 오류 (timeout, 파싱 실패, 인증 실패) 는 즉시 SLEEP 송신 후 종료

**호출 사이트 변경** (`app/master/m_main.c`):
- `Master_PollImplantData`: `CC1101_SendPacketLBT_Agile(...)` → `Master_TxLBT_SameChannel(...)`
- `Master_WriteToImplant`: `CC1101_SendPacketLBT(g_active_channel, ...)` → `Master_TxLBT_SameChannel(...)`
- `Master_SendSleepCommand`: `CC1101_SendPacketLBT(g_active_channel, ...)` → `Master_TxLBT_SameChannel(...)`

**전파 이득**:
- Slave 는 **항상 비컨에서 받은 채널만 감시** → 동기화 손실 제로
- Master 는 채널 N 일시 점유 상황에서도 짧은 백오프로 재시도 → 같은 세션을 유지
- 채널 N 이 영구 점유되면 다음 Wake-up 에서 새 채널을 선택 → 장기적으로도 안정적

변경 파일 (4):
`common/protocol.h`, `app/master/m_main.c`, `app/slave/s_main.c`, `docs/20230710.md`

### 11.4 PHY 변조 정합화 상세 (CC1101 2-FSK / CC2500 OOK)

#### 배경 — 설계 의도와 코드 불일치

초기 레지스터 프로파일은 두 칩 모두 `MDMCFG2 = 0x13` (MOD_FORMAT=`001`, **GFSK**) 으로 세팅되어 있었으나, 프로젝트 설계 의도는 다음과 같이 달랐다.

| 칩 | 설계 의도 | 이전 코드 (틀림) | 이번 패치 (정합) |
|----|-----------|-----------------|-----------------|
| CC1101 (400 MHz MICS) | **2-FSK** (MICS 권장, 단순 변조) | GFSK `0x13` | **2-FSK `0x03`** |
| CC2500 (2.4 GHz WOR) | **OOK/ASK** (WOR 감도·전력 최적화) | GFSK `0x13` | **OOK `0x33`** |

#### 변경 내용

**CC2500 (`cc2500/main/cc2500.c`, `CC2500_InitWakeUp26MHz` / `CC2500_InitMICSLike26MHz`):**

| 레지스터 | 이전 | 이후 | 근거 |
|----------|------|------|------|
| `MDMCFG2` | `0x13` (GFSK) | `0x33` (OOK/ASK, 30/32 sync) | MOD_FORMAT 001→011 |
| `DEVIATN` | `0x15` (~5.16 kHz) | `0x00` | OOK 에서 하드웨어가 DEVIATN 을 무시 |
| `FREND0` | `0x10` (PA_POWER=0) | `0x11` (PA_POWER=1) | 2-엔트리 PA ramp 사용 |
| `PATABLE` | 1 B (`0xFE`) | 2 B `{0x00, 0xFE}` | `[0]`='0' OFF, `[1]`='1' ON |

**CC1101 (`cc1101/main/cc1101.c`, `CC1101_InitMICSLike26MHz`):**

| 레지스터 | 이전 | 이후 | 근거 |
|----------|------|------|------|
| `MDMCFG2` | `0x13` (GFSK) | `0x03` (2-FSK, 30/32 sync) | MOD_FORMAT 001→000 |
| `DEVIATN` | `0x35` | `0x35` (유지) | 2-FSK 에서 유효 (~20.63 kHz) |

**문서 변경 (`docs/20230710.md`):**

- §3.x `CC1101_InitMICSLike26MHz()` 설명: "GFSK, ~50 kbps" → "2-FSK, ~50 kbps"

- §4.3.10 CC1101 vs CC2500 비교표: `GFSK 50 kbps` → `2-FSK 50 kbps` (CC1101), `GFSK 2.4 kbps` → `OOK 2.4 kbps` (CC2500)
- §4.3.11 **MDMCFG2 레지스터 비트맵 참조** 신설 — 비트 레이아웃, MOD_FORMAT 코드표, 본 프로젝트 실제 설정값

#### 호환성 · 체크리스트

- PHY 설정 변경이므로 **구버전 펌웨어와 무선 호환 없음** — Master / Slave 양쪽을 동시에 본 버전으로 업데이트해야 한다.

# MICS 체내삽입형 의료기기 통신 시스템 설계문서

> 작성일: 2023-07-10
> MCU: STM32U575 Q-series
> RF: CC1101 (400 MHz MICS), CC2500 (2.4 GHz Wake-up)

---

## 1. 시스템 개요

체내삽입형 의료기기(Implant)와 체외 프로그래머(Programmer) 간 무선 통신 시스템.

| 구분 | 역할 | 설명 |
|------|------|------|
| **Master (Programmer)** | 체외 장치 | 통신 개시, LBT 수행, 채널 선택 |
| **Slave (Implant)** | 체내 삽입형 | 수동 응답만, 초저전력 대기 |

### 듀얼 라디오 아키텍처

| 라디오 | 주파수 | 용도 |
|--------|--------|------|
| CC2500 | 2.4 GHz ISM | Wake-up 전용 (WOR 모드) |
| CC1101 | 402~405 MHz MICS | 데이터 통신 (LBT 필수) |

---

## 2. 통신 흐름

```
Master (Programmer)                         Slave (Implant)
    |                                           |
    |  [CC1101] LBT -> 빈 MICS 채널 N 사전선정   |
    |                                           |
    |  [CC2500] Wake-up Beacon (x10 반복)       |  MCU: Stop2, CC2500: WOR
    |  payload = DEVICE_ID + SESS + CHANNEL(N)  |
    |------------------------------------------>|
    |                                           |  GDO0 EXTI -> MCU 기상
    |                                           |  SESS/CHANNEL(N) 저장
    |           [CC2500] Wake-up ACK            |
    |           (DEVICE_ID+SESS+CH echo)        |
    |<------------------------------------------|
    |                                           |  CC1101 wake + ch=N 튜닝
    |  --- CC1101 400MHz MICS 데이터 세션 ---   |
    |                                           |
    |  [CC1101] POLL_REQ (ch=N, SESS+DEVID)     |
    |------------------------------------------>|
    |           [CC1101] POLL_RESP (데이터)      |
    |<------------------------------------------|
    |                                           |
    |  [CC1101] DATA_WRITE (설정값)             |
    |------------------------------------------>|
    |           [CC1101] DATA_ACK               |
    |<------------------------------------------|
    |                                           |
    |  [CC1101] SLEEP_CMD                       |
    |------------------------------------------>|
    |           [CC1101] SLEEP_ACK              |
    |<------------------------------------------|
    |                                           |  CC1101: Sleep
    |                                           |  CC2500: WOR 재진입
    |                                           |  MCU: Stop2 복귀
```

### 3.1 빌드 환경: STM32CubeIDE 1.19.0

본 프로젝트는 **STM32CubeIDE 1.19.0** 으로 빌드한다 (CMake/Makefile 미사용).
워크스페이스 안에 다음 두 프로젝트를 생성한다:

| 프로젝트명 | 진입점 | HW |
|----------|--------|-----|
| `MICS_Master_Programmer` | `app/master/main.c` | 체외 Programmer 보드 |
| `MICS_Slave_Implant`     | `app/slave/main.c`  | 체내삽입형 Implant 보드 |

공통 절차 (양 프로젝트 공히):

1. **새 프로젝트 생성**: `File → New → STM32 Project` → Part Number `STM32U575xx`
2. **`.ioc` 페리페럴 설정** (상세는 3.2/3.3 참조)
3. **Generate Code** → `Core/`, `Drivers/` 자동 생성
4. **공통 라이브러리 Linked Folder 추가**:
   ```
   Project Explorer → New → Folder → Advanced → Link to alternate location
     PARENT-2-PROJECT_LOC/cc1101/main   →  cc1101_src
     PARENT-2-PROJECT_LOC/cc2500/main   →  cc2500_src
     PARENT-2-PROJECT_LOC/common         →  common_src
   ```
   > `PARENT-2-PROJECT_LOC` = 워크스페이스 기준 두 단계 위 (리포지토리 루트).
   > 워크스페이스가 리포지토리 루트와 같다면 `PARENT-1-PROJECT_LOC` 사용.
5. **Include Path 추가** (Properties → C/C++ Build → Settings → MCU GCC Compiler → Include paths):
   ```
   ../../cc1101/header
   ../../cc2500/header
   ../../common
   ```
6. **`Core/Src/main.c`** 의 USER CODE 영역에서 `app/<role>/main.c` 의 진입점 호출
   (CubeMX가 매번 덮어쓰는 영역 외부에 코드 배치)
7. **Build All (Ctrl+B)** → `Debug/MICS_<role>.elf` 산출

> 두 프로젝트는 **같은 공통 코드를 공유**하므로, `cc1101.c` 등을 수정하면
> 다음 빌드 시 양쪽 모두 새로 빌드된다.

### 3.2 Master (Programmer) 프로젝트 셋업 상세

체외 장치(Programmer) 펌웨어 프로젝트 (`MICS_Master_Programmer`).

#### 3.2.1 새 프로젝트 생성

1. STM32CubeIDE 실행 → **File → New → STM32 Project**
2. **Target Selector**:
   - Series: `STM32U5`
   - Part Number: `STM32U575xx` (Q-series가 사용하는 정확한 부품번호 선택)
3. **Project Setup**:
   - Project Name: `MICS_Master_Programmer`
   - Targeted Language: `C`
   - Targeted Binary Type: `Executable`
   - Targeted Project Type: `STM32Cube`
4. **Firmware Library**: 최신 STM32CubeU5 HAL 패키지 사용 (예: 1.5.0 이상)
5. **Finish** — `.ioc` 파일과 기본 `Core/`, `Drivers/` 폴더가 생성된다.

#### 3.2.2 .ioc (CubeMX) 페리페럴 설정

`MICS_Master_Programmer.ioc` 더블클릭:

**클럭 (RCC)**
- HSE: Disable (저전력용 MSI만 사용)
- LSE: Crystal/Ceramic Resonator (RTC용 32.768 kHz)
- SYSCLK: MSI 4 MHz (저전력 동작 시) ~ 최대 80 MHz
- PLL은 사용하지 않거나 옵션

**SPI1 (CC1101용)**
- Mode: Full-Duplex Master
- Baud Rate: ~5 MHz (CC1101 최대 10 MHz)
- Clock Polarity (CPOL): Low
- Clock Phase (CPHA): 1 Edge
- NSS: Disable (수동 GPIO 제어)
- Pin: PA5(SCK), PA6(MISO), PA7(MOSI)

**SPI2 (CC2500용)**
- 동일한 설정으로 SPI2 활성화
- Pin: PB13(SCK), PB14(MISO), PB15(MOSI)

**GPIO**
| 핀 | 모드 | 설명 |
|----|------|------|
| PA4 | Output Push-Pull, High | CC1101 CS |
| PB12 | Output Push-Pull, High | CC2500 CS |
| PB0 | Input, Pull-Down | CC1101 GDO0 (선택: EXTI) |
| PB1 | EXTI Mode with Rising trigger | CC2500 GDO0 (Wake-up용) |
| PC13 | Output Push-Pull | LED 표시 (선택) |

**RTC**
- Activate Clock Source ✔
- Activate Calendar ✔
- WakeUp ✔ (RTC Wake-Up Timer)
- Clock Source: LSE

**NVIC**
- EXTI line 1 (PB1, CC2500 GDO0) 인터럽트 활성화
- RTC Wake-Up Timer 인터럽트 활성화

설정 완료 후 **Project → Generate Code** (또는 Ctrl+S)

#### 3.2.3 main.c 통합

CubeMX가 매번 `Core/Src/main.c`를 덮어쓰므로, 아래 방식을 사용한다:

- `app/master/main.c` 의 `int main(void)` → `void Master_AppMain(void)` 로 변경
- 내부 `HAL_Init()` 호출은 제거 (CubeMX가 이미 호출)
- CubeMX `Core/Src/main.c` 의 USER CODE 영역에서 호출:

```c
/* USER CODE BEGIN Includes */
extern void Master_AppMain(void);
/* USER CODE END Includes */

/* USER CODE BEGIN 2 */
Master_AppMain();   /* never returns */
/* USER CODE END 2 */
```

#### 3.2.4 빌드 & 다운로드

1. **Project → Build All** (Ctrl+B)
2. **Run → Debug As → STM32 C/C++ Application** (또는 ST-Link로 플래시)

빌드 산출물: `Debug/MICS_Master_Programmer.elf`

### 3.3 Slave (Implant) 프로젝트 셋업 상세

체내삽입형 디바이스(Implant) 펌웨어 프로젝트 (`MICS_Slave_Implant`).
**초저전력 최적화가 필수**이므로 Master와 설정이 일부 다르다.

#### 3.3.1 새 프로젝트 생성

1. **File → New → STM32 Project**
2. Part Number: `STM32U575xx`
3. Project Name: `MICS_Slave_Implant`
4. 나머지는 Master 프로젝트와 동일. **Finish**

#### 3.3.2 .ioc (CubeMX) 페리페럴 설정 — 초저전력 최적화

**클럭 (저전력 우선)**
- HSE: **Disable** (절대 사용 금지)
- HSI16: Disable (필요 시에만)
- LSE: Crystal Resonator 32.768 kHz (RTC + Stop2 wake-up용)
- SYSCLK Source: **MSI**
- MSI Range: **4 MHz** (Run 시) — 더 낮은 1 MHz도 가능
- Voltage Scaling: **Range 4** (최저 전압, Stop2 진입 가능)

**SPI**
- **SPI1** (CC1101): Run 중에만 활성, 슬립 시 De-Init
  - Pin: PA5(SCK), PA6(MISO), PA7(MOSI)
- **SPI2** (CC2500): WOR 모드에서는 OFF (CC2500 자체가 알아서 RX 함)
  - Pin: PB13(SCK), PB14(MISO), PB15(MOSI)

**GPIO (Stop 2 누설 최소화 필수)**
| 핀 | 모드 | 설명 |
|----|------|------|
| PA4 | Output Push-Pull, High | CC1101 CS |
| PB12 | Output Push-Pull, High | CC2500 CS |
| PB0 | Input, No Pull (또는 Analog) | CC1101 GDO0 |
| PB1 | **EXTI Mode with Rising trigger, Pull-Down** | CC2500 GDO0 → MCU Wake-up |
| 사용하지 않는 핀 | **Analog** | Stop 2 누설 최소화 |

**RTC (주기적 자가점검)**
- Activate Clock Source ✔
- Activate Calendar ✔
- **WakeUp ✔** (60초 주기)
- Clock Source: **LSE** (32.768 kHz)

**NVIC (Wake-up 소스)**
- **EXTI line 1** (PB1, CC2500 GDO0) → Enable, 우선순위 높게
- **RTC Wake-Up Timer** → Enable
- SysTick: Wake-up 후에만 Resume (저전력 모드 진입 시 Suspend)

**Power 설정**
- CubeMX에서 직접 설정 어려움 → 코드에서 `LP_Init()` 호출 시 자동 설정:
  - `HAL_PWREx_EnableUltraLowPowerMode()`
  - `HAL_PWREx_DisableFastWakeUpFromStop()`
  - Voltage Scale 4

설정 완료 → **Project → Generate Code**

#### 3.3.3 빌드 최적화 설정 (Implant 필수)

**Properties → C/C++ Build → Settings → MCU GCC Compiler**:

| 항목 | 값 | 이유 |
|------|----|----|
| Optimization Level | `-Os` (Optimize for size) | 코드 크기 ↓, Flash 누설 ↓ |
| Debug Level | `Minimal (-g1)` | 릴리즈 빌드 |
| Function sections | ✔ `-ffunction-sections` | 미사용 함수 제거 |
| Data sections | ✔ `-fdata-sections` | 미사용 데이터 제거 |

**MCU GCC Linker → General**:
- ✔ `-Wl,--gc-sections` (Garbage collect unused sections)

**Build Configuration**: `Release` 사용 권장.

#### 3.3.4 main.c 통합

Master와 동일한 방식으로 `app/slave/main.c` 를 통합:

- `int main(void)` → `void Slave_AppMain(void)` 로 변경
- 내부 `HAL_Init()` 제거

```c
/* Core/Src/main.c 내부 */
/* USER CODE BEGIN Includes */
extern void Slave_AppMain(void);
/* USER CODE END Includes */

/* USER CODE BEGIN 2 */
Slave_AppMain();   /* never returns */
/* USER CODE END 2 */
```

#### 3.3.5 EXTI 콜백 연결

`Core/Src/stm32u5xx_it.c` 의 `EXTI1_IRQHandler()` 가 자동 생성되며,
`HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin)` 콜백을 호출한다.
`app/slave/main.c` 에 정의된 `HAL_GPIO_EXTI_Callback()` 이 약한 심볼을 오버라이드한다.

#### 3.3.6 빌드 & 플래시

1. **Project → Build All** (Release 설정)
2. **Run → Run As → STM32 C/C++ Application**

빌드 산출물: `Release/MICS_Slave_Implant.elf` (또는 `.hex`/`.bin`)

> 💡 **저전력 측정**: 디버거를 분리하고 외부 배터리로 동작시켜야 정확한 ~3.7 uA가 측정됨.
> ST-Link 연결 상태에서는 SWD 라인 누설로 수십 uA 더 측정될 수 있음.

## 4. 모듈별 기능 상세

### 4.1 CC1101 드라이버 (`cc1101/`)

400 MHz MICS 대역 데이터 통신 담당.

| 함수 | 설명 |
|------|------|
| `CC1101_InitMICSLike26MHz()` | MICS 대역 레지스터 초기화 (402.15 MHz base, GFSK, ~50 kbps) |
| `CC1101_SendPacketLBT()` | LBT(Listen Before Talk) 후 패킷 전송 |
| `CC1101_SendPacketLBT_Agile()` | **채널 어질리티 + 랜덤 백오프** 지원 LBT 전송 |
| `CC1101_WaitAndReadPacket()` | 타임아웃 기반 패킷 수신 대기 |
| `CC1101_FindFreeChannel()` | 10개 MICS 채널 스캔하여 빈 채널 탐색 |
| `CC1101_ChannelClear()` | CCA(Clear Channel Assessment) |
| `CC1101_EnterSleep()` | CC1101 SPWD 슬립 모드 진입 |
| `CC1101_WakeFromSleep()` | CS 토글로 슬립 해제 |
| `CC1101_SetPATable()` | TX 출력 전력 설정 |

**MICS 채널 계획:**
- Base: 402.150 MHz
- 채널 간격: ~300 kHz
- 채널 수: 10개 (ch0~ch9: 402.15 ~ 404.85 MHz)
- LBT 임계값: -80 dBm
- 최대 백오프: 50 ms
- 최대 재시도: 8회

### 4.2 CC2500 드라이버 (`cc2500/`)

2.4 GHz ISM 대역 Wake-up 전용.
**드라이버는 프로토콜 비의존** — MAC PDU 조립은 애플리케이션 계층(`common/protocol.h`)이 담당하고, 드라이버는 바이트 스트림만 송수신함.

| 함수 | 설명 |
|------|------|
| `CC2500_InitWakeUp26MHz()` | WOR 최적화 설정 (~2.4 kbps, 높은 수신 감도) |
| `CC2500_EnterWOR()` | Wake-on-Radio 모드 진입 (~500 ms 스니프 주기) |
| `CC2500_ConfigureWOR()` | WOR 타이밍 파라미터 설정 |
| `CC2500_SendWakeUpBeacon()` | **미리 조립된 MAC PDU**(pkt, pkt_len)를 N회 반복 전송 |
| `CC2500_SendPacketDirect()` | LBT 없이 직접 패킷 전송 |
| `CC2500_WaitAndReadPacket()` | 타임아웃 기반 패킷 수신 대기 |
| `CC2500_EnterSleep()` | CC2500 슬립 모드 |
| `CC2500_WakeFromSleep()` | 슬립 해제 |

**WOR 설정:**
- 데이터 레이트: ~2.4 kbps (WOR 감도 최적화)
- 스니프 주기: ~500 ms
- Sync word: 0xBEEF (데이터 통신과 구분)
- GDO0: CRC OK 시 assert -> EXTI 인터럽트

### 4.3 프로토콜 (`common/protocol.h`)

#### 4.3.1 공통 애플리케이션 PDU (CC1101 전 구간 + CC2500 Proto_BuildPacket 구간)

```
+------+------+------+------+------------------+------+
| LEN  | FCF  | SEQ  | SESS | PAYLOAD          | CRC  |
| 1B   | 1B   | 1B   | 1B   | 0..58B (no FEC)  | 2B   |
|      |      |      |      | 0..105B (w/FEC)  |      |
+------+------+------+------+------------------+------+
      └─────── MAC HEADER (3B) ─────┘
```

- **LEN / CRC**: CC1101·CC2500 하드웨어가 자동 처리 (PKTCTRL0.LENGTH_CONFIG=variable, CRC_EN=1)
- **MAC 프레임은 두 라디오 공통** — PHY(주파수·변조·싱크워드)만 다를 뿐 PDU 구조는 동일
- **DEVICE_ID는 MAC 헤더에 없음** — SESS(1B)가 세션 식별 담당. DEVICE_ID는 Wake-up 비컨 PAYLOAD 내부에만 사용 (SESS 할당 전 단계이므로)

| 필드 | 크기 | 설명 |
|------|------|------|
| LEN | 1 B | 프레임 길이 (헤더 포함, 최대 63 B) — 라디오 자동 |
| FCF | 1 B | 프레임 제어 필드 (비트 구조 참조) |
| SEQ | 1 B | 시퀀스 번호 (0~255 wrap) |
| SESS | 1 B | 세션 ID (0x01~0xFE 할당, 0x00=미할당, 0xFF=브로드캐스트) |
| PAYLOAD | 0~58 B (FEC 미적용) / 0~105 B (FEC 적용) | 데이터 페이로드 |
| CRC | 2 B | 라디오 자동 |

#### 4.3.2 FCF (Frame Control Field) 비트 구조

| 비트 | 매크로 | 설명 |
|------|--------|------|
| Bit 7 (0x80) | `MICS_FCF_ACK_REQ` | ACK 요청 플래그 (1: ACK 필요, 0: ACK 불필요) |
| Bit 6 (0x40) | `MICS_FCF_FRAGMENT` | Fragment 패킷 플래그 (1: Fragment, 0: 일반) |
| Bit 5 (0x20) | Reserved | 예약 |
| Bit 4 (0x10) | Reserved | 예약 |
| Bit 3~0 (0x0F) | 프레임 타입 | `MICS_FCF_TYPE_DATA` (0x00), `MICS_FCF_TYPE_ACK` (0x01), `MICS_FCF_TYPE_COMMAND` (0x02) |

#### 4.3.3 Payload 타입 코드 (PAYLOAD[0] @ DATA/COMMAND frame)

| 코드 | 매크로 | 설명 |
|------|--------|------|
| 0x10 | `MICS_DTYPE_ECG_DELTA` | ECG 델타 압축 데이터 |
| 0x20 | `MICS_DTYPE_BATTERY` | 배터리 상태 |
| 0x30 | `MICS_DTYPE_TEMP` | 온도 데이터 |
| 0x40 | `MICS_DTYPE_STATUS_LOG` | 상태 로그 |
| 0xA0 | `MICS_DTYPE_COMMAND` | 제어 명령 |
| 0xFF | `MICS_DTYPE_EMERGENCY` | 긴급 데이터 (우선 전송) |

#### 4.3.4 Command 서브코드 (PAYLOAD[1] @ DTYPE_COMMAND frame)

| 코드 | 매크로 | 방향 | 라디오 | 설명 |
|------|--------|------|--------|------|
| 0x01 | `CMD_WAKEUP_REQ` | M→S | CC2500 | Wake-up 비컨 (DEVICE_ID+SESS 포함) |
| 0x02 | `CMD_WAKEUP_ACK` | S→M | CC2500 | Wake-up 응답 (DEVICE_ID+SESS 에코) |
| 0x10 | `CMD_POLL_REQ` | M→S | CC1101 | 데이터 요청 |
| 0x11 | `CMD_POLL_RESP` | S→M | CC1101 | 데이터 응답 (또는 DATA 프레임 직접 응답) |
| 0x20 | `CMD_DATA_WRITE` | M→S | CC1101 | 설정값 쓰기 |
| 0x21 | `CMD_DATA_ACK` | S→M | CC1101 | 쓰기 확인 |
| 0x30 | `CMD_SLEEP_CMD` | M→S | CC1101 | 슬립 명령 |
| 0x31 | `CMD_SLEEP_ACK` | S→M | CC1101 | 슬립 확인 |
| 0x40 | `CMD_KEEPALIVE` | M→S | CC1101 | 세션 연장 |
| 0x41 | `CMD_KEEPALIVE_ACK` | S→M | CC1101 | 세션 연장 확인 |

#### 4.3.5 Wake-up 비컨 PAYLOAD 레이아웃 (CC2500, 2.4 GHz)

```
Offset  Bytes  Field
------  -----  ----------------------------------------
  0      1    MICS_DTYPE_COMMAND (0xA0)
  1      1    CMD_WAKEUP_REQ (0x01)  or  CMD_WAKEUP_ACK (0x02)
  2..5   4    DEVICE_ID (target or responder)
  6      1    SESS    (Master가 할당한 세션 ID)
  7      1    CHANNEL (Master가 LBT로 사전 선정한 MICS 채널 0..9)
  8      1    flags / reserved
                                      총 PAYLOAD 길이 = 9 B
```

- MAC SESS 필드 = `PROTO_SESS_BROADCAST` (0xFF) — 세션이 아직 성립되지 않았으므로
- 할당된 SESS는 PAYLOAD offset 6에 실림 → Slave가 기억 → 이후 MICS 통신에서 MAC SESS로 사용
- CHANNEL(offset 7)은 Master가 **비컨 송신 직전**에 400 MHz LBT로 찾은 빈 MICS 채널
  - Slave는 해당 채널로 **바로 튜닝**해서 대기 → 10개 채널 스캔 없음 → 임플란트 전력 절감
  - Slave는 ACK에 동일한 채널을 **에코**하여 Master가 채널 동기화를 확인
  - 미할당 시 `PROTO_CHANNEL_UNASSIGNED` (0xFF) — 권장되지 않음

#### 4.3.6 400 MHz COMMAND 프레임 PAYLOAD 레이아웃 (FCF type=0x02)

```
Offset  Bytes  Field
------  -----  ----------------------------------------
  0      1    MICS_DTYPE_COMMAND (0xA0)
  1      1    sub-command (CMD_POLL_REQ, CMD_DATA_WRITE, ...)
  2..5   4    DEVICE_ID  ← 2.4 GHz 웨이크업에서 넘어온 Implant ID
  6..    N    command-specific arguments
                                      헤더 고정 = 6 B
```

- DEVICE_ID(offset 2..5)는 **방향과 무관하게 항상 Implant(Slave)의 ID**
- Master → Slave: "나는 네가 2.4 GHz에서 응답한 implant 0xXXXX와 통신 중이다"
- Slave → Master: "나는 2.4 GHz에서 응답한 그 implant 0xXXXX가 맞다"

#### 4.3.7 400 MHz DATA 프레임 PAYLOAD 레이아웃 (FCF type=0x00)

```
Offset  Bytes  Field
------  -----  ----------------------------------------
  0      1    data type (MICS_DTYPE_ECG_DELTA, _BATTERY, _TEMP, ...)
  1..4   4    DEVICE_ID  ← Implant의 자기 ID (송신자 증명)
  5..    N    telemetry / data body
                                      헤더 고정 = 5 B
```

#### 4.3.8 2.4 GHz → 400 MHz 식별자 핸드오프

**Master 및 Slave 양측이 모든 400 MHz 패킷에서 다음 두 조건을 **모두** 검사:**

| 검증 항목 | 필드 | 크기 | 속도 | 역할 |
|---|---|---|---|---|
| 세션 필터 | MAC.SESS | 1 B | 빠름 | 빠른 1차 필터 |
| 신원 검증 | payload.DEVICE_ID | 4 B | O(4) | 2.4 GHz 핸드오프 바인딩 |

```
[2.4 GHz wake-up]                    [400 MHz MICS session]
    M ──── DEVICE_ID + SESS ────> S        M ─── SESS + DEVICE_ID ───> S
    M <─── DEVICE_ID + SESS ──── S        M <── SESS + DEVICE_ID ──── S
         │                                      │          │
         └── 2.4 GHz에서 공유 ──────────────────┴──────────┘
                 ↓
           양측 모두 수신 패킷의
           SESS == g_session_id     ← 필터링 (1B)
           DEVICE_ID == expected    ← 핸드오프 검증 (4B)
```

- **Master**: 수신 패킷의 `payload.DEVICE_ID == g_target_id` 확인
- **Slave**: 수신 패킷의 `payload.DEVICE_ID == g_my_id` 확인
- 어느 쪽이든 불일치 시 **silent drop** (무응답) — 스푸퍼/오착신 방어
- 헬퍼: `Proto_VerifyPayloadDeviceID(pkt, offset, expected)` — COMMAND는 offset=2, DATA는 offset=1

#### 4.3.9 CC1101 vs CC2500 MAC 동일성

| 레이어 | CC1101 (400 MHz MICS) | CC2500 (2.4 GHz Wake-up) | 공통? |
|--------|----------------------|-------------------------|-------|
| PHY (변조/레이트) | GFSK 50 kbps, MICS ch0~9 | GFSK 2.4 kbps, ch0 | ❌ |
| Sync Word | 0xD391 | 0xBEEF (의도적 구별) | ❌ |
| 하드웨어 LEN/CRC | PKTCTRL0 variable+CRC_EN | 동일 | ✅ |
| **MAC PDU** | FCF\|SEQ\|SESS\|PAYLOAD | 동일 | ✅ **Yes** |
| Builder/Parser | `Proto_BuildPacket`/`Proto_ParsePacket` | 동일 | ✅ |

**결론**: 두 라디오는 PHY만 다르고 MAC PDU는 완전히 동일. 단일 `Proto_Packet` 구조와 단일 빌더/파서가 양쪽을 모두 처리함.

## 5. Master (Programmer) 기능 구조

```
master_programmer.c
│
├── Master_InitHardware()
│   ├── CC1101_InitMICSLike26MHz()    // 400 MHz 데이터 라디오
│   ├── CC2500_InitWakeUp26MHz()      // 2.4 GHz 웨이크업 라디오
│   └── LP_Init()                     // 저전력 인프라
│
├── Master_CommunicationSession()
│   │
│   ├── [Phase 1] Master_WakeUpImplant()
│   │   ├── CC2500_SendWakeUpBeacon()     // 비컨 10회 반복 전송
│   │   └── CC2500_WaitAndReadPacket()    // WAKEUP_ACK 대기
│   │
│   ├── [Phase 2] Master_PollImplantData()
│   │   ├── CC1101_SendPacketLBT_Agile()  // LBT + 채널 어질리티
│   │   │   ├── CC1101_FindFreeChannel()  // 빈 채널 탐색
│   │   │   ├── CC1101_SendPacketLBT()    // CCA 후 전송
│   │   │   └── Random Back-off           // 실패 시 랜덤 대기
│   │   └── CC1101_WaitAndReadPacket()    // POLL_RESP 수신
│   │
│   ├── [Phase 2] Master_WriteToImplant()
│   │   ├── CC1101_SendPacketLBT()        // DATA_WRITE 전송
│   │   └── CC1101_WaitAndReadPacket()    // DATA_ACK 수신
│   │
│   └── [Phase 3] Master_SendSleepCommand()
│       ├── CC1101_SendPacketLBT()        // SLEEP_CMD 전송
│       └── CC1101_WaitAndReadPacket()    // SLEEP_ACK 수신 (선택)
│
└── Radios -> Sleep
```

**LBT + 채널 어질리티 알고리즘:**
1. 10개 MICS 채널 순차 스캔 (CCA)
2. RSSI < -80 dBm 인 채널 발견 시 -> 해당 채널에서 전송
3. 모든 채널 사용중 -> 랜덤 백오프 (1~50 ms)
4. 최대 8회 재시도
5. 재시도 초과 시 `CC1101_ERR_CCA` 반환

```
slave_implant.c
│
├── Slave_InitHardware()
│   ├── CC2500_InitWakeUp26MHz()      // WOR 최적화 설정
│   ├── CC1101_EnterSleep()           // 400 MHz 라디오 슬립
│   └── LP_Init() + LP_SetVoltageScaling_LowPower()
│
├── [메인 루프] while(1)

│   ├── [GDO0 기상 시] Slave_HandleWakeUp()
│   │   ├── CC2500_ReadPacket()       // 비컨 패킷 읽기
│   │   ├── Device ID 검증            // 내 ID 또는 브로드캐스트
│   │   ├── SESS 저장 (g_session_id)
│   │   ├── CHANNEL 저장 (g_active_channel)  // Master가 LBT로 선정한 MICS ch
│   │   └── CC2500_SendPacketDirect() // WAKEUP_ACK (DEVICE_ID+SESS+CHANNEL 에코)
│   │
│   └── [ACK 성공 시] Slave_HandleMICSSession()
│       ├── CC1101_WakeFromSleep() + InitMICSLike26MHz()
│       │
│       ├── [세션 루프] g_active_channel 로 직접 튜닝 (스캔 없음)
│       │   ├── CC1101_WaitAndReadPacket(ch=g_active_channel) // 100ms 윈도
│       │   ├── SESS 필터 + DEVICE_ID 검증 (핸드오프)
│       │   ├── Slave_ProcessCommand()       // 명령 처리
│       │   │   ├── POLL_REQ  -> 센서 데이터 응답
│       │   │   ├── DATA_WRITE -> 설정 저장 + ACK
│       │   │   ├── SLEEP_CMD -> 세션 종료
│       │   │   └── KEEPALIVE -> 세션 타이머 리셋
│       │   └── CC1101_SendPacketLBT()       // 응답 전송
│       │
│       └── [세션 종료] CC1101_EnterSleep()
│           └── -> 메인 루프 복귀 (DeepSleep)
```

**Slave 핵심 원칙:**
- **스스로 통신을 개시하지 않음** - 오직 Master의 요청에 응답
- **채널 스캔 없음** - Master가 LBT로 선정한 채널(비컨 offset 7)으로 바로 튜닝 → 전력 절감
- **세션 타임아웃** - 10초 무응답 시 자동 슬립 복귀

## 7. 전력 소비 예상

> **단위 표기 규칙**
> - **순시 소비전류 (Instantaneous current)**: `uA` / `mA` — 해당 모드에서 흐르는 전류
> - **시간 평균 소비 (Time-averaged consumption)**: `uA·h` / `mAh` — 1시간 누적 사용량
> - "통신 세션 중 ~20 mA" 는 **세션이 활성화된 동안의 순시 전류**이며,
>   1시간 내내 이 값이 흐른다는 의미가 아님 (`mAh` 아님).

### 7.1 Slave (Implant) - 일반 대기 상태 (주 동작 모드)

| 구성요소 | 모드 | 순시 소비전류 |
|----------|------|--------------:|
| STM32U575 | Stop 2 | ~2 uA |
| CC2500 | WOR (~500ms 주기 sniff) | ~1.5 uA 평균 |
| CC1101 | SPWD (Sleep) | ~0.2 uA |
| **합계 (Idle)** | **대기 시** | **~3.7 uA** |

### 7.2 Slave (Implant) - 통신 세션 중 (피크 전류, 단발성)

| 구성요소 | 모드 | 순시 소비전류 |
|----------|------|--------------:|
| STM32U575 | Run (MSI 4 MHz) | ~2 mA |
| CC1101 | RX / TX | ~15~17 mA |
| CC2500 | Idle (세션 중 OFF 가능) | ~1.5 mA |
| **합계 (Active 피크)** | **세션 진행 중에만** | **~18~20 mA** |

> ⚠️ 위의 ~20 mA 값은 **세션이 진행 중일 때의 순시(peak) 전류**입니다.
> 1시간 동안 흐르는 양(`mAh`)이 아닙니다.

### 7.3 Slave (Implant) - 시간 평균 소비량 (1시간 기준, mAh)

세션 길이와 빈도에 따른 1시간당 실제 소비 누적량 예시:

| 세션 길이 | 세션 빈도 | 활성 시간/시간 | Active 누적 | Idle 누적 | **총 소비/시간** |
|----------:|----------:|---------------:|------------:|----------:|-----------------:|
| 0.5 s | 1회/시간 | 0.5 s (0.014%) | 0.0028 mAh | 0.0037 mAh | **~0.0065 mAh/h** |
| 1.0 s | 1회/시간 | 1.0 s (0.028%) | 0.0056 mAh | 0.0037 mAh | **~0.0093 mAh/h** |
| 1.0 s | 6회/시간 | 6.0 s (0.17%) | 0.033 mAh | 0.0037 mAh | **~0.037 mAh/h** |
| 1.0 s | 60회/시간 | 60 s (1.67%) | 0.33 mAh | 0.0037 mAh | **~0.34 mAh/h** |

> 예: **세션 1초/시간 사용 시 ≈ 0.0093 mAh/h ≈ 0.22 mAh/일 ≈ 6.7 mAh/월**
> CR2032 (~225 mAh) 기준 약 **2~3년 수명 가능** (자가방전 미고려).

### 7.4 Master (Programmer / External Device) - 전력 소비

체외 장치는 일반적으로 **유선 충전 또는 대용량 배터리** 사용. Implant만큼 엄격한 저전력 제약 없음.

#### 7.4.1 Master - Idle (사용자 트리거 대기)

| 구성요소 | 모드 | 순시 소비전류 |
|----------|------|--------------:|
| STM32U575 | Stop 2 (또는 LP Run) | ~2 uA ~ 50 uA |
| CC2500 | SPWD (Sleep) | ~0.4 uA |
| CC1101 | SPWD (Sleep) | ~0.2 uA |
| 사용자 인터페이스 (LCD/버튼) | 표시 OFF | ~10 uA ~ 1 mA |
| **합계 (Idle)** | **대기 시** | **~12 uA ~ 1 mA** |

#### 7.4.2 Master - Wake-up Beacon 전송 중 (CC2500)

| 구성요소 | 모드 | 순시 소비전류 |
|----------|------|--------------:|
| STM32U575 | Run (MSI 4 MHz) | ~2 mA |
| CC2500 | TX (+1 dBm, 비컨 반복) | ~21 mA |
| CC1101 | Sleep | ~0.2 uA |
| **합계 (Beacon TX 피크)** | **비컨 송출 중에만** | **~23 mA** |

> 비컨 1회 송출 시간: ~5 ms × 10회 반복 + 간격 = **~150 ms 정도**

#### 7.4.3 Master - LBT + MICS 데이터 통신 중 (CC1101)

| 구성요소 | 모드 | 순시 소비전류 |
|----------|------|--------------:|
| STM32U575 | Run (MSI 4 MHz) | ~2 mA |
| CC1101 | RX (LBT/CCA) | ~15~17 mA |
| CC1101 | TX (-16 dBm, MICS 한도) | ~13~14 mA |
| CC2500 | Sleep | ~0.4 uA |
| **합계 (LBT/RX 피크)** | **CCA/RX 중** | **~17~19 mA** |
| **합계 (TX 피크)** | **데이터 송신 중** | **~15~16 mA** |

#### 7.4.4 Master - 세션 평균 소비량 (1시간 기준)

| 시나리오 | 세션 시간 | 세션 빈도 | **총 소비/시간** |
|----------|----------:|----------:|-----------------:|
| 의사 진료 중 (활성) | 5 s | 12회/시간 | ~0.32 mAh/h |
| 외래 모니터링 (저빈도) | 2 s | 1회/시간 | ~0.011 mAh/h + Idle |
| 연속 데이터 수집 (고빈도) | 1 s | 60회/시간 | ~0.34 mAh/h + Idle |

> Master는 일반적으로 **300~2000 mAh 배터리** 또는 **USB 전원** 사용 → 수일~수개월 운용 가능.

### 7.5 비교 요약

| 항목 | Slave (Implant) | Master (Programmer) |

## 8. MICS 규정 준수 사항

- **LBT (Listen Before Talk)**: Master는 전송 전 반드시 CCA 수행
- **최대 TX 출력**: MICS 대역 25 uW (-16 dBm) EIRP
- **채널 점유 시간**: 최소화 (짧은 패킷, 빠른 세션)
- **채널 어질리티**: 채널 사용중 시 랜덤 백오프 + 채널 전환
- **Slave 수동성**: 체내 기기는 스스로 전송 개시 금지

### 10.1 f019e91 상세 (MAC PDU 재설계)

- **이전 (6B 헤더)**: `LEN | CMD | SEQ | DEVICE_ID[4] | PAYLOAD(≤55) | CRC`
- **이후 (3B 헤더)**: `LEN | FCF | SEQ | SESS | PAYLOAD(≤58 no-FEC / ≤105 FEC) | CRC`
- **FCF 비트 구조 도입**: ACK_REQ(Bit7) / FRAGMENT(Bit6) / FrameType(Bit3-0: DATA/ACK/COMMAND)
- **Payload 타입 코드**: `MICS_DTYPE_ECG_DELTA` (0x10), `BATTERY` (0x20), `TEMP` (0x30), `STATUS_LOG` (0x40), `COMMAND` (0xA0), `EMERGENCY` (0xFF)
- **SESS 기반 주소지정**: Master가 Wake-up 시 1~254 rolling 할당 → Slave가 기억 → MICS 세션 전 구간 `Proto_MatchSession()` 으로 필터링
- **DEVICE_ID(4B)**: MAC 헤더에서 제거, Wake-up 비컨 PAYLOAD offset 2~5 에만 포함
- **드라이버 계층 정리**: `CC2500_SendWakeUpBeacon(dev, device_id, id_len, repeat)` → `CC2500_SendWakeUpBeacon(dev, pkt, pkt_len, repeat)` — 드라이버에서 PDU 조립 코드 제거, MAC PDU 조립은 `Proto_BuildPacket`/`Proto_BuildBeaconPayload` 가 단독 담당
- **MAC 단일화 확인**: CC1101과 CC2500은 PHY만 다르고 (주파수·변조·싱크워드) MAC PDU와 빌더·파서는 완전히 동일함을 §4.3.9 표에 명문화

### 10.2 비컨 채널 프리할당 상세

**문제**: Master가 LBT로 빈 MICS 채널을 선정해도 Slave에게 알려줄 방법이 없으면,
Slave는 10개 채널(0~9)을 반복 스캔해야 함 → 임플란트 배터리 낭비.

**해결 방법 1**: 비컨 payload를 1바이트 확장해서 선정 채널을 실어 보냄.

**프로토콜 변경**:
- `PROTO_BEACON_PAYLOAD_LEN` 8 → 9
- `PROTO_BEACON_CHANNEL_OFFSET = 7` (신규)
- `PROTO_BEACON_FLAGS_OFFSET` 7 → 8
- `PROTO_CHANNEL_UNASSIGNED = 0xFF` (신규 — 미할당 마커)
- `Proto_BuildBeaconPayload()` 시그니처에 `uint8_t channel` 인자 추가

**Master 동작 변경** (`app/master/main.c` `Master_WakeUpImplant()`):
1. **비컨 송신 *전*** 에 `CC1101_FindFreeChannel()` 호출
2. 반환된 채널을 `g_active_channel` 에 저장
3. `Proto_BuildBeaconPayload(..., channel=g_active_channel, ...)` 로 비컨에 포함
4. ACK 수신 시 `payload[7] == g_active_channel` 확인 → 불일치 시 세션 거부

**Slave 동작 변경** (`app/slave/main.c`):
- `g_active_channel` 상태 변수 추가 (초기값 `PROTO_CHANNEL_UNASSIGNED`)
- `Slave_HandleWakeUp()`: 비컨 payload[7] 범위 검증 후 `g_active_channel` 저장, ACK에 에코
- `Slave_HandleMICSSession()`: `for (ch=0..9)` 스캔 루프 제거, `g_active_channel` 단일 채널로 직접 튜닝
- 세션 종료 시 `g_active_channel` 을 `PROTO_CHANNEL_UNASSIGNED` 로 초기화

**전력 이득**: Slave가 채널당 50 ms × 10개 = 500 ms/루프 스캔 → 단일 채널 100 ms 윈도 (~5배 절감).

`common/protocol.h`, `app/master/main.c`, `app/slave/main.c`, `docs/20230710.md`
